# 新建集群中部分监控面板缺失

## 现象
- 新建集群的集群概览中，`apiserver`、`etcd` 等监控面板与老集群不一致，部分监控面板缺失。
- 现场对比发现，这并非面板渲染异常，而是相关监控指标本身未被新集群暴露或采集。
- 问题与 `enhancer` 插件安装状态及其带来的监控链路差异直接相关。

## 排查过程与命令
- 首先对比新旧集群的插件差异，确认新集群未安装 `Alauda Container Platform Cluster Enhancer`，因此缺少由 `enhancer` 提供的相关监控指标与面板。
- 进一步确认后发现，安装 `Alauda Container Platform Cluster Enhancer` 虽可补齐缺失指标，但当前版本同时存在 `vmselect` 资源异常占用的已知问题，因此处理不能只停留在“安装插件”，还需要联动处理相关告警规则。
- 为避免直接修改无法回退，先备份相关 `alertrule`：
  ```bash
  kubectl get alertrule -n cpaas-system cpaas-certificates-rule -oyaml > /root/certificatealertrule.yaml
  ```
- 随后删除 `cpaas-certificates-rule`，避免证书相关告警继续影响 `vmselect` 资源占用：
  ```bash
  kubectl delete alertrule -n cpaas-system cpaas-certificates-rule
  ```
- 再通过 `vmui` 查询确认 `enhancer` 实际会暴露以下三类关键指标：
  - `certificateexpirestime`
  - `certificateexpiresstatus`
  - `etcdbackuprecords`
- 进一步梳理指标与面板的映射关系后，确认：
  - `certificateexpirestime`、`certificateexpiresstatus` 会影响平台组件证书过期告警以及 `Certificates built-in / Certificates` 监控面板；
  - `etcdbackuprecords` 会影响 `etcd Backup / etcd 备份` 监控面板。
- 如果不需要保留上述指标采集，则可进一步禁用 `enhancer` 采集链路。操作前先备份 `servicemonitor`：
  ```bash
  kubectl get servicemonitor -n cpaas-system cluster-enhancer-apiserver -oyaml > /root/enhancer-servicemonitor.yaml
  ```
- 然后删除对应 `servicemonitor`：
  ```bash
  kubectl delete servicemonitor -n cpaas-system cluster-enhancer-apiserver
  ```
- 综合以上排查结果可以确认，本次问题的根本原因是新集群未安装 `enhancer`，导致相关指标与面板缺失；而在补齐 enhancer 后，又需要同时考虑该版本中 `vmselect` 已知资源占用问题，因此需要配套调整告警规则与指标采集范围。

## 根因与修复方案
- **根因**
  - 新集群未安装 `enhancer` 插件，导致部分由 `enhancer` 提供的监控指标和监控面板不存在。
  - 同时当前版本存在 `vmselect` 资源异常占用的已知问题，安装 enhancer 后若不处理相关规则和采集配置，可能引入新的资源压力。
- **临时缓解方案**
  - 安装 `enhancer` 插件以补齐缺失监控面板。
  - 删除 `cpaas-certificates-rule`，避免证书告警影响 `vmselect`。
  - 若不需要保留相关指标采集，可删除 `cluster-enhancer-apiserver` 对应 `servicemonitor`，减少采集压力。
- **根本解决方案**
  - 在新集群初始化阶段统一校验并补齐 `enhancer` 依赖，确保监控面板与老集群保持一致。
  - 针对 `v4.2` 中 `vmselect` 资源异常占用的已知问题，建立与 `enhancer` 指标采集联动的默认处理策略，避免安装补齐指标后再次引入监控资源问题。
