# ceph 池容量在平台上显示偏大

## 现象
- `ceph` 池已使用容量在平台上显示偏大，与实际使用情况不符。
- 现场查看实际使用未到 `50%`，但平台容量条显示接近满盘。
- 该问题属于前端展示口径问题，而不是底层存储真实写满。

## 排查过程与命令
- 首先查看 `ceph-cluster` 详情，并执行：
  ```bash
  ceph -s
  ```
  状态为 `HEALTH_WARN`，存在 `suspicious read errors` 和 `recently crashed` 的 daemon。
- 随后执行：
  ```bash
  ceph df detail
  ```
  确认池实际使用量并未接近满盘。
- 在页面侧查看存储池容量条，确认平台展示的是逻辑容量口径，而非直接映射物理使用率。
- 进一步核对已用容量的查询逻辑：
  ```text
  sum by()(cephpoolstored * on(poolid,instance) groupleft(name) cephpoolmetadata{vmcluster="集群",name="pool名"})
  ```
  该查询拿到的是存储池的已使用逻辑存储数据量。
- 再核对可用容量的查询逻辑：
  ```text
  sum by()(cephpoolmaxavail * on(poolid, instance) groupleft(name) cephpool_metadata{ vmcluster = "集群名", name = "pool名" })
  ```
  该查询拿到的是存储池剩余可用逻辑存储数据量，也就是理论上 OSD 还能写入的最大逻辑数据量。
- 综合页面展示与查询口径可以确认，前端容量条使用的是逻辑容量口径，因此在逻辑口径下看起来接近满盘，但实际物理空间并未满。

## 根因与修复方案
- **根因**
  - 平台前端容量展示口径使用了已使用逻辑存储量和理论最大可写入逻辑量。
  - 由于逻辑容量与实际物理使用不一致，导致 `ceph` 池看起来接近满盘，但实际空间并未满。
- **临时缓解方案**
  - 以 `ceph df detail` 等实际使用量结果为准，避免仅凭前端容量条判断池是否已满。
- **根本解决方案**
  - `v4.3` 已修复该前端显示问题。
  - `v4.2` 后续发版小版本会修复。
