# 节点重复加入集群导致 NotReady

## 现象
- 节点 `10.71.136.171` 在当前集群中处于 `NotReady`。
- `kubelet` 报错：
  ```text
  Unable to authenticate the request due to invalid bearer token, square/go-jose: error in cryptographic primitive
  ```
- `describe node` 显示 `Ready`、`MemoryPressure`、`DiskPressure`、`PIDPressure` 均为 `Unknown`，原因是 `NodeStatusUnknown`，信息为 `Kubelet stopped`，并带有：
  - `node.kubernetes.io/unreachable:NoExecute`
  - `node.kubernetes.io/unreachable:NoSchedule`

## 排查过程与命令
- 首先查看节点状态，确认 `10.71.136.171` 在当前集群中已处于 `NotReady`，且多个节点条件均为 `Unknown`。这表明问题并非某个单项资源压力异常，而是节点状态上报链路整体出现问题。
- 随后检查 `kubelet` 客户端证书：
  - `notBefore=Oct 28 02:05:01 2025 GMT`
  - `notAfter=May 4 12:51:00 2036 GMT`
  
  证书本身未发现过期异常，因此可以排除“证书到期导致认证失败”这一方向。
- 继续检查 `apiserver` 和 `docker`，未发现明显异常报错，说明控制面与容器运行时并没有直接暴露出同类故障信号。
- 在节点侧查看 `10250` 连接，发现存在 `10.71.136.171:10250` 与其他节点的连接记录，说明该节点上的 kubelet 并非完全未运行，而是其认证或归属行为存在异常。
- 为进一步确认是否存在集群归属冲突，在另一个集群中执行：
  ```bash
  kubectl get node | grep 136.171
  ```
  结果发现同一 `IP` `10.71.136.171` 在另一个集群中状态为 `Ready`。
- 由此可以确认：两个集群中同时存在同一个 `IP` 对应的节点。该节点的 `kubelet` 状态实际上已上报到另一个集群，导致当前集群中的节点对象失去正常心跳更新，最终表现为 `NotReady`。
- 清理重复加入的节点后，节点状态恢复正常。综合整个排查过程可以确认，本次故障并非证书失效或单点进程崩溃，而是节点重复加入多个集群造成的归属冲突。

## 根因与修复方案
- **根因**
  - 同一 `IP` 节点 `10.71.136.171` 被同时加入两个集群。
  - 节点上的 `kubelet` 状态上报到另一个集群，导致当前集群中的节点对象无法获得正常状态更新。
  - 最终表现为当前集群中该节点 `NotReady`，并出现 `invalid bearer token` 相关报错。
- **临时缓解方案**
  - 清理重复加入的节点，确保同一节点只归属于一个集群。
- **根本解决方案**
  - 建立节点纳管前的唯一性校验机制，避免同一 `IP` 或同一宿主机被重复加入多个集群。
  - 在节点注册与回收流程中增加交叉集群检查，提前识别节点归属冲突并阻断重复接入。
