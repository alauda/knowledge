# ACP 中间件与数据服务 FAQ

> 这页优先做**一线分诊入口**：先按问题类型跳到对应口径，再决定是答 FAQ、看 checklist，还是转 runbook。

## 0. 首页分诊 / 快速跳转

- **先问 3 个最小信息**：版本 / 组件 / 场景（支持咨询、开启配置、还是现网排障）

- **问“改了为什么不生效”**：先看本页 5、8

- **问 RabbitMQ 创建 / 删除 / 插件 / 协议**：先看本页 3、9

## 1. Redis / Kafka / RabbitMQ 这类数据服务，客户问“支不支持某环境/某协议/某部署方式”，先怎么答？
先别直接答支持或不支持，先判断客户问的是哪一类：

- 环境支持：例如纯 IPv6、global 集群 / 业务集群是否都支持
- 协议 / 插件支持：例如 RabbitMQ 是否支持 MQTT 插件
- 方案边界：例如 Kafka 能不能帮业务把两个相同上游无脑并到一个 topic
- 现网异常伪装成支持咨询：表面问“支不支持”，实际是在问“为什么现在不生效”

先问什么：
- 产品版本
- 涉及组件（Redis / Kafka / RabbitMQ / Nacos）
- 问的是平台托管实例能力，还是开源组件原生能力
- 目标环境（IPv4 / IPv6 / 双栈、global / 业务集群）
- 是新部署咨询、迁移切换，还是现网实例排障

不要先承诺什么：
- 不要把“单个组件能用”直接承诺成“整套环境全链路都支持”
- 不要把“开源能做到”直接承诺成“平台默认支持并负责运维”

关联案例：
- [TICKET-1333207424](../../../ticket-documents/cases/TICKET-1333207424%204.1.3版本ACP%20数据服务redis、kafka、rabbitmq%20是否支持纯v6%20环境.md)
- [TICKET-1291455564](../../../ticket-documents/cases/TICKET-1291455564%20kafka相关咨询.md)
- 参考清单：[ACP Database & Middleware Support Boundary Checklist](../../../knowledge_base/troubleshooting/database-middleware-support-boundary-checklist.md)

## 2. 客户问“4.1.3 的 Redis / Kafka / RabbitMQ 支不支持纯 IPv6，global 和业务集群是不是都支持”，现场怎么回？
先判断什么：
- 这是环境支持边界问题，不是单纯组件功能问题
- 客户问的是单组件支持，还是整套数据服务在 global + 业务集群上的完整部署边界

先问什么：
- 明确 ACP 版本
- 明确涉及的是 Redis、Kafka、RabbitMQ 哪些组件
- 明确要覆盖 global 集群、业务集群，还是两者都要
- 明确是新部署咨询，还是已有实例准备迁移到纯 IPv6

不要先承诺什么：
- 不要只回一句“支持纯 v6”就结束
- 不要在没确认集群范围前，把单组件结论外推成 global 和业务集群全部都支持

更稳的话法：

> 这类问题先按版本、组件范围和集群形态确认支持边界；尤其要区分是单组件可部署，还是 global 集群和业务集群整套链路都要覆盖，再给结论会更稳。

关联案例：
- [TICKET-1333207424](../../../ticket-documents/cases/TICKET-1333207424%204.1.3版本ACP%20数据服务redis、kafka、rabbitmq%20是否支持纯v6%20环境.md)

## 3. 客户问“dataservice 的 RabbitMQ 实例支不支持 MQTT 插件，端口怎么开、怎么用”，先怎么接？
先判断什么：
- 客户问的是平台数据服务里的托管 RabbitMQ，还是自建开源 RabbitMQ
- 问的是“插件理论能不能开”，还是“平台是否提供明确开启方式、端口和运维支持”

先问什么：
- 目标插件是 `rabbitmq_mqtt`、`rabbitmq_web_mqtt`，还是别的插件
- 现场是否已经有平台侧 RabbitMQ 实例
- 除了插件本身，是否还在追问端口、开启入口、使用文档、后续谁负责支持

不要先承诺什么：
- 不要把“开源 docker-compose 能配插件”直接等同于“平台托管实例可按同样方式操作”
- 不要在没确认平台文档、入口和支持范围前，先承诺插件开启路径和端口方案

更稳的话法：

> 先区分您问的是开源 RabbitMQ 原生能力，还是平台托管 RabbitMQ 的支持范围。开源侧能启插件，不等于平台托管实例默认按同样方式开放；如果还涉及端口、开启方式和后续运维支持，需要继续按平台侧文档和支持边界确认。

关联案例：
- [TICKET-1291886194](../../../ticket-documents/cases/TICKET-1291886194%20dataservice的rabbitmq实例是否支持mqtt插件.md)

## 4. 客户问“两个一模一样的数据源，能不能直接并到 Kafka 一个 topic，让下游无脑订阅”，先怎么答？
先判断什么：
- 这更像业务方案边界，不是单纯“Kafka 支不支持”
- 客户真正想要的是上游故障切换时，下游不感知且不处理重复数据

先问什么：
- 两个上游数据源是否真的会同时写入
- 下游是否能接受重复数据、做幂等、做去重
- 客户要的是平台现成功能，还是希望平台兜底业务高可用设计

不要先承诺什么：
- 不要先承诺“Kafka 能帮你天然合并成一个无脑 topic”
- 不要默认把重复数据、幂等治理、切换策略算成平台侧承诺范围

更稳的话法：

> 这类通常不是简单的组件支持问题，而是业务架构边界问题。如果两个完全相同的数据源同时写同一个 topic，默认就会面对重复数据；下游要不要感知、谁负责去重和幂等，需要先说清楚。

关联案例：
- [TICKET-1291455564](../../../ticket-documents/cases/TICKET-1291455564%20kafka相关咨询.md)

## 5. Nacos 调整数据库连接到新的库，通常先确认什么？
先判断什么：
- 这是单纯改连接参数，还是伴随切库 / 迁移 / 新租户切换
- 现场说的“不生效”是真配置变更没生效，还是其实混进了连接症状 / 外部接入 / 生命周期问题
- 如果用户描述里同时出现“外部应用连不上”“Web UI 正常”“SDK 报错”，先不要急着按 config-change 处理
- 现场说的“不生效”是改错对象、没改全，还是改完没重启

先问什么：
- 数据库连接信息到底落在哪个配置对象（如 ConfigMap / Secret）
- 现场是否同时有 `nacos-cm`、`nacos-conf` 这类容易混淆的对象，谁是源、谁是衍生，是否存在“改了一个对象但实际生效源是另一个”的可能
- 支持过程中是否已经出现过“这两个会自动同步”与“看现场可能两个都要改”这类前后不一致说法；如果出现，先停止口述判断，回到现场核对真实生效源
- 地址、端口、用户名、密码、数据库名 / 租户是否都变了
- 新库是否已准备好必要对象（库、用户、表、权限）
- 修改后是否已经重启并验证生效
- 如果中途回退过，当前实例是否仍带着旧值或命中旧连接，相关衍生对象是否也已同步回退

不要先承诺什么：
- 不要一上来就说“改个 ConfigMap 就行”
- 不要在没确认配置来源和重启要求前，先承诺改完会自动同步、自动生效
- 不要因为对象名里带数据库配置，就默认它一定是实际生效源
- 不要在没核对回退/残留值前，就把“还连着旧库”直接解释成平台没应用新配置
- 不要把未验证过的迁移切换方案直接当成标准动作承诺

更稳的话法：

> 这类问题先别急着改参数，先确认连接信息实际落在哪个真实生效对象、相关字段是否改全、对象值和运行态是否一致、改完是否需要重启，以及新库对象是否准备完整；如果现场还做过回退，也要继续核对当前实例是否仍带着旧值，否则很容易出现“看起来改了，其实没真正生效”。

关联案例：
- [TICKET-1331278694](../../../ticket-documents/cases/TICKET-1331278694%20nacos调整ob数据库，调整到新的数据库.md)
- 参考清单：[ACP Database & Middleware Config Change Checklist](../../../knowledge_base/troubleshooting/database-middleware-config-change-checklist.md)

## 6. Nacos 外部应用访问异常，但 Web UI / ALB 看起来又正常，先看什么？
先不要一上来就归成“网络不通”，这类问题很容易是 **Nacos 客户端接入条件** 没对齐。

先判断什么：
- 是浏览器能打开 Web UI，但应用侧注册/发现失败
- 还是应用通过 ALB / NodePort 访问时，端口映射和客户端参数不匹配
- 还是客户端 SDK 与当前 Nacos 版本不兼容

先问什么：
- 当前 Nacos 版本是多少
- 客户端 SDK / Spring Cloud Alibaba 版本是多少
- 用的是 ALB、NodePort，还是别的暴露方式
- 外部暴露端口是否保持了 `8848 / 9848` 这组约束关系
- 用官方/平台提供的 demo 是否能正常连接

不要先承诺什么：
- 不要先把“Web UI 能打开”直接等同于“应用侧一定能正常注册和发现”
- 不要把“外部访问异常”直接粗归成网络设备或 ALB 故障
- 不要在没核对端口映射和 SDK 兼容性前，就先判平台侧组件异常

更稳的话法：

> 这类问题先别只按“外部访问不通”处理。若 Web UI 可访问，但应用侧 Nacos 注册/发现异常，更常见根因是客户端配置、SDK 兼容性，或外部暴露端口没有满足 Nacos 的端口映射约束；建议先用 demo 验证，再核对客户端参数和端口关系。

关联案例：
- [TICKET-1279061724](../../../ticket-documents/cases/TICKET-1279061724%20nacos外部访问.md)

## 7. Redis 客户端连接数高，但看不清来自哪里，怎么办？
如果只靠默认 `client list` 看不清客户端来源，往往需要在应用侧补更清晰的客户端标识。

较稳的思路是：
- 在应用侧为客户端连接增加 `ClientName`
- 用 hostname 或业务标识区分连接来源

否则连接数高的时候，只看默认信息很难快速定位是哪一批客户端造成的。

关联案例：
- [TICKET-1281905874](../../../ticket-documents/cases/TICKET-1281905874%20dataservice%20redis的连接客户端问题.md)

## 7b. 客户问“Redis 该给什么规格、什么模板”，现场先收集什么？
如果客户还没进入故障排查，而是在问 **“我们有 Redis 需求，应该怎么选规格、模板、访问方式”**，不要先直接回一个实例大小。

先判断什么：
- 这是故障工单，还是方案/选型咨询
- 客户要的是缓存用途、持久化用途，还是分片高并发场景
- 同时还在问“应用怎么连”“分片地址是不是都要配”时，要把访问方式和部署模式一起回答

先问什么：
- 业务场景是什么
- 预估总数据量（GB）
- 高峰期 QPS
- 是否要求高可用（自动故障切换）
- 部署模式偏好（单实例 / 哨兵 / 集群）
- Redis 版本要求
- 是否要求持久化
- 最大连接数
- 网络与访问方式（内网 / 外网 / 是否经 LB 暴露）

不要先承诺什么：
- 不要在没问清数据量、QPS、连接数前，先拍一个规格
- 不要把“访问示例”与“部署模式”拆开单答
- 不要默认所有场景都能套同一模板

更稳的话法：

> Redis 规格与模板建议要先看业务场景、容量、QPS、高可用、部署模式、持久化和访问方式；这些前提没问清之前，直接给实例大小很容易失真。建议先按标准采集项把需求收全，再决定推荐的模板和客户端访问示例。

关联案例：
- [TICKET-1349525984](../../../ticket-documents/cases/TICKET-1349525984%20数据服务-Redis.md)

## 8. Redis 改了规格又回退，为什么还可能起不来？
这类问题不要只盯着“现在想改成多少”，更要看 **namespace limit、对象值、运行态和值班现场看到的值是否已经重新一致**。

先判断什么：
- 是否经历过 `调大 -> 再回退` 的变更过程
- 当前失败是 namespace limit 卡住，还是 STS/运行态仍残留旧值
- operator 是否已经真正把最新目标值重新收敛下去

先问什么：
- 本次 Redis 规格从多少改到多少，后来又回退到多少
- 当前命名空间容器资源 limit 允许的上限是多少
- STS 当前资源字段显示的值是多少
- 实际运行对象 / Pod 看到的值是否和界面目标值一致
- 是否在调整 limit 后还需要重启 operator 才恢复

不要先承诺什么：
- 不要看到界面已经改回去，就默认 STS 和运行态也已经回到新值
- 不要只看到 namespace limit 不够，就认为根因已经找全
- 不要把“回退后还起不来”直接理解成最新目标值本身有问题

更稳的话法：

> 这类问题要把链路看完整：先确认目标规格改动历史，再核对 namespace limit、STS 当前实际值，以及运行态是否仍残留旧值；如果 namespace limit 已经改回或放开，但 STS / 运行态还没重新收敛一致，即使界面已经改回去，实例也仍可能起不来。

关联案例：
- [TICKET-1324229924](../../../ticket-documents/cases/TICKET-1324229924%20redis无法正常启动.md)

## 9. MQ / RabbitMQ 创建慢、创建失败、删除不了，先看什么？
先分三类：
- 真正创建失败
- 创建很慢但最终能起来
- 控制面状态判断异常，导致删除/健康状态异常

这类问题别只盯业务实例本身，通常还要往 operator 和状态收敛逻辑看；如果重启 operator 仍未恢复，还要继续看实例本身是否存在镜像模式、rebalance 或数据一致性问题。

先问什么：
- 前台报的是创建失败、删除失败，还是健康状态异常
- CR / 事件里是否出现 `ready.allReplicasReady` 这类健康状态求值报错
- pod 是真的起不来，还是只是启动 / 变更非常慢
- operator 资源是否紧张，是否尝试过独立命名空间或更高资源的 operator
- 是否开启了镜像模式、MQTT，或已经出现数据不一致 / rebalance 相关迹象
- 当前更像控制面收敛慢，还是实例侧状态已经异常到需要处理镜像策略 / CR 注解

不要先承诺什么：
- 不要把前台“创建失败”直接等同于 RabbitMQ 实例本身一定失败
- 不要把“重启 operator 无效”直接收敛成平台无解
- 不要忽略 RabbitMQ CR 注解、镜像策略与实例状态之间的联动
- 不要只因为 operator 参与了流程，就把实例侧数据一致性问题都归成控制面故障
- 不要因为现场说过“改过参数/重启过 operator”，就直接把问题改判成 RabbitMQ 参数变更未生效；若没有明确参数对象和旧值残留证据，先按生命周期收敛异常处理更稳

更稳的话法：

> 这类问题先区分是真失败、收敛很慢，还是健康状态判断先报错；如果 operator 重启后仍未恢复，再继续看实例本身是否存在镜像模式、rebalance 或数据不一致问题。也就是说，要同时判断“控制面是否收敛慢”和“实例侧是否已经异常”，这样比把它们都粗归成“MQ 创建失败”更稳。

关联案例：
- [TICKET-1315422524](../../../ticket-documents/cases/TICKET-1315422524%20无法创建新mq集群以及删除掉mq集群.md)
- [TICKET-1282552714](../../../ticket-documents/cases/TICKET-1282552714%20rabbitmq%20状态异常.md)

## 10. MongoDB 这类组件在页面里“不能直接部署”，先看什么？
先不要一上来就当平台故障，有时这类现象更像 **产品方案入口或上架方式本来就有限制**。

先判断什么：
- 客户说的是“页面里没有直接部署入口”
- 还是“有入口但实际部署失败”
- 当前组件是否本来就需要按单独方案、单独上架方式交付

先问什么：
- 当前 ACP 版本是多少
- 现场说的 MongoDB 是页面想直接部署，还是已有独立交付方案
- 这次诉求是功能咨询、产品支持边界确认，还是现网部署失败排障

不要先承诺什么：
- 不要一看到“页面无法部署”就直接按故障受理
- 不要在没确认产品交付方式前，先承诺页面一定存在通用部署入口
- 不要把“可支持交付”直接等同于“可在当前页面路径直接自助部署”

更稳的话法：

> 这类问题先区分是“部署入口本来就不同”还是“已有入口但实际失败”。如果该组件需要按单独方案上架或交付，就不应直接按页面故障理解，先把产品支持边界和交付入口确认清楚会更稳。

关联案例：
- [TICKET-1290447044](../../../ticket-documents/cases/TICKET-1290447044%204.0版本mongodb页面无法部署.md)

## 11. MongoDB / 中间件实例内存使用率很高，先看什么？
先不要只盯着“90%+”这个数字，有些中间件实例高内存占用不一定立刻等于故障，更关键的是判断：**这是短时波峰、正常工作集占用，还是规格确实不够**。

先判断什么：
- 是持续高内存，还是短时峰值
- 业务侧最近有没有流量变化、数据量变化
- 当前诉求是“先确认是否异常”，还是已经准备直接扩容

先问什么：
- 监控时间范围有没有拉长到 7 天左右
- MongoDB 实例当前规格是多少
- 最近有没有明显流量增长、数据增长、查询模式变化
- 现场是想先确认是否需要扩容，还是已经验证过必须调大规格

不要先承诺什么：
- 不要一看到 90%+ 就直接给出“马上扩容”的结论
- 不要在没看趋势前，把瞬时高点当成长期异常
- 不要把“能改 YAML 调大规格”直接等同于“根因已经确认”

更稳的话法：

> 这类问题先把监控时间拉长，区分是持续高占用还是短时波峰，再结合当前实例规格和业务变化判断是否真的需要扩容；否则很容易把工作集正常占用直接当成故障。

关联案例：
- [TICKET-1287821974](../../../ticket-documents/cases/TICKET-1287821974%20mongodb%20内存使用率高.md)

## 12. Redis 单实例想配到 16G / 32G，现场先怎么答？
这类问题先别只答“能不能填”，要先区分客户问的是：**页面能否配置、更大规格是否建议、还是业务容量已经逼着必须这么做**。

先判断什么：
- 当前是哨兵模式还是 Cluster 模式
- 客户是在问“平台支持边界”，还是在问“最佳实践建议”
- 业务容量诉求是单实例变大，还是更适合走分片

先问什么：
- 当前 Redis 架构（哨兵 / Cluster）
- 当前单实例规格和目标规格
- 当前是否启用了持久化
- 调大内存后，存储是否也会同步放大
- 是否还要求强制反亲和 / 高可用拓扑约束

不要先承诺什么：
- 不要把“页面能配更大”直接等同于“官方推荐这么配”
- 不要在没确认持久化、RDB/AOF 加载时长、反亲和约束前，就先承诺单实例继续放大没问题
- 不要忽略 `maxmemory`、磁盘容量与重启恢复时间这几个连带约束

更稳的话法：

> 这类问题先分“是否能配”与“是否建议这样配”。即使平台侧能调大规格，也不代表单实例越大越好；还要结合 Redis 架构、持久化加载时间、存储容量、反亲和约束和分片方案一起判断。

关联案例：
- [TICKET-1267631304](../../../ticket-documents/cases/TICKET-1267631304%20dataservice内redis支持的最大内存大小.md)

## 11. Redis 改大内存后反而起不来，先看什么？
这类问题别先当成“Redis 本身坏了”，更常见的是：**实例目标规格已经超过命名空间容器资源上限，或者 operator/STS 状态没有及时收敛**。

先判断什么：
- 这是改规格后首次拉起失败
- 还是已经改回较小值，但实例对象里仍残留旧资源值
- 现场异常点是在 Pod 事件、STS 资源字段，还是 operator 收敛链路

先问什么：
- Redis 之前从多少内存改到多少
- 当前命名空间的容器内存上限是多少
- `describe pod` / 事件里是否明确报资源上限冲突
- STS 当前实际资源值是否仍停留在旧的大规格
- operator 是否重启过、是否在重启后开始恢复

不要先承诺什么：
- 不要一看到“Redis 起不来”就先往数据损坏或镜像异常上靠
- 不要只把 Redis 规格改回去就认为一定会自动恢复
- 不要忽略命名空间 limit 与 operator 状态收敛这两个约束点

更稳的话法：

> 这类问题先核对 Redis 目标规格有没有超过命名空间容器资源上限，再看 STS 当前资源值和 operator 状态是否已经收敛；否则表面像 Redis 启动失败，实质是规格调整链路没有闭环。

关联案例：
- [TICKET-1324229924](../../../ticket-documents/cases/TICKET-1324229924%20redis无法正常启动.md)

## 12. 升级后 mysql-mgr 有个实例起不来，现场先看什么？
这类问题别先只按“升级问题”处理。标题虽然常带“升级后”，但更稳定的核心往往是：**MGR 集群成员异常、脑裂后无法自愈、rejoin 失败，最后表现成某个实例一直起不来**。

先判断什么：
- 是单个 MGR 成员起不来，还是整个集群状态已经异常
- 现场是卡在 Pod 层、MySQL 实例层，还是 MGR 集群状态层
- 之前是否已经尝试过 `rejoinInstance`、`removeInstance/addInstance`、`dba.rebootClusterFromCompleteOutage()`

先问什么：
- 当前 MGR 集群整体状态是什么
- 起不来的成员是哪一个，是否是单点问题还是集群级问题
- 现场有没有明确脑裂 / complete outage 迹象
- 当前是重加入失败，还是整个集群无法正常恢复
- 之前执行过哪些 mysqlsh 修复动作，执行结果是什么
- 如果已经做过 `rejoinInstance` / `force rejoin` / `removeInstance/addInstance` 仍失败，要继续确认这是不是其实已经越过“单成员故障”阶段，进入了完整集群修复场景

容易混淆但不要先吸进来的：
- **迁移咨询**：例如节点迁移 / TopoLVM 迁移方案里提到 `mysql_mgr`，核心更可能是存储或迁移路径，而不是 MGR 成员修复
- **别的 MySQL 方案状态异常**：例如 PXC 被停用后前台仍显示 processing，这不等于 MGR 脑裂或 rejoin 失败

不要先承诺什么：
- 不要一看到“升级后”就先把根因归到升级流程本身
- 不要把单个成员起不来直接等同于普通 Pod/容器重启问题
- 不要在没确认 MGR 集群状态前，就先承诺只重启实例或删 Pod 能恢复

更稳的话法：

> 这类问题先区分是“升级触发后暴露出的 MGR 集群异常”，还是普通实例启动问题。若已经出现脑裂、rejoin 失败或 complete outage，通常要按 MGR 集群修复路径处理，而不是只在 Pod 层反复重启。

关联案例：
- [TICKET-1315953644](../../../ticket-documents/cases/TICKET-1315953644%20升级后问题：mysql-mgr有个实例起不来.md)

## 13. MySQL MGR 的 mysql_bin_log 磁盘使用量过高，先看什么？
这类问题别先只按“磁盘满了”处理。更稳定的核心通常是：**业务写入变化、binlog 保留策略、复制状态与 PVC 容量规划** 叠加后，最终表现成 binlog 目录增长过快。

先判断什么：
- 当前是真的 binlog 堆积，还是业务写入量阶段性放大
- MGR 集群是否运行正常、是否存在复制异常
- 告警的本质是参数保留过长，还是当前容量本身已经不够

先问什么：
- 当前 MGR 集群整体状态是否正常
- 各节点数据量、binlog 目录占用量大概是多少
- 最近业务是否有批量加字段、批量写入或其它明显变更
- 当前 `binlog-expire-logs-seconds` 设置为多少
- 当前 PVC 容量与预计保留周期是否匹配
- 是否已经粗算过“单日 binlog 增量 × 保留天数”大概会占掉多少空间
- 如果现场只提“升级后”“突然告警”，也要继续追问最近是否刚做过批量 DDL、字段新增或业务写入放量，否则很容易把容量问题误判成版本或实例异常

不要先承诺什么：
- 不要一看到 binlog 告警就先默认是复制异常
- 不要只建议清理 binlog，而不先确认保留周期和业务写入背景
- 不要在没核算容量与保留周期关系前，就先承诺只改参数或只扩容一定能解决
- 不要把“集群正常”直接等同于“容量就没问题”；正常运行也可能只是把容量压力慢慢累积出来

更稳的话法：

> 这类问题先分“binlog 是异常堆积”还是“保留策略与业务写入变化叠加后的容量压力”。如果集群本身运行正常，就应优先结合写入变化、binlog 保留时长和 PVC 容量一起判断，是先扩容、先缩短保留周期，还是两者都要做。

关联案例：
- [TICKET-1343215994](../../../ticket-documents/cases/TICKET-1343215994%20甘肃响泉口生产环境：edge命名空间的mysql-mgr告警mysql_bin_log磁盘使用量过高.md)

## 14. Backup Center / 外部 S3 / 恢复路径问题，先怎么分？
先别把这类问题都笼统叫“备份有问题”，统一运维里至少要先分四层：

- 备份入口问题：实例有没有备份、最近一次结果如何、存储位置在哪
- 外部 S3 注册问题：信息是否填全、网络是否可达、连通性检查是否通过
- 恢复路径问题：是恢复到新实例，还是误以为所有组件都支持恢复到已有实例
- 平台版本阶段问题：是不是 ACP v3→v4 过渡期里 global/workload 版本不一致，导致 Data Services 视图异常

先问什么：
- 当前组件是什么（MGR / Kafka / RabbitMQ / Redis ...）
- 是看不到备份、备份失败、恢复入口不符合预期，还是页面显示异常
- 如果是外部存储，Endpoint / AK / SK / Bucket 是否完整
- 外部存储是否对当前环境网络可达
- 如果是恢复到已有实例，当前组件是不是 MGR
- 当前 global 和 workload cluster 版本是否一致

不要先承诺什么：
- 不要把 Backup Center 理解成某个组件自己的私有页面
- 不要默认所有组件都支持恢复到已有实例
- 不要把 Data Services 页面异常直接判成实例故障

更稳的话法：

> 这类问题先分成备份中心、外部 S3、恢复路径和平台版本阶段边界四层来判断会更稳；很多“备份/页面异常”并不是单组件故障，而是 unified-om 能力边界或版本阶段问题。

关联资料：
- [ACP Database & Middleware Unified O&M Quick Card](../notes/database-middleware-unified-om-quick-card.md)

## 15. Inspection Center 相关问题，先怎么接？
先不要把 inspection 当成“看状态页”。这条线更像任务化巡检：
- 定义巡检任务 `Inspection`
- 执行生成结果对象 `InspectionJob`
- 结果根据触发入口显示在实例页、任务中心或 report 视图

先问什么：
- 这是想创建巡检任务，还是想查某次巡检结果
- 目标 component、namespace、instances 是什么
- `trigger` 是 `Manual` 还是 `Schedule`
- 如果是 `Schedule`，`schedule` 是否已按 UTC 理解和填写
- 结果是从实例详情页触发，还是从任务中心触发
- `InspectionJob.status.results` 里的 `healthyStatus / message` 是什么

不要先承诺什么：
- 不要把 `Inspection` 任务定义和 `InspectionJob` 执行结果混成一个对象
- 不要默认按北京时间写 crontab 就会按北京时间执行
- 不要看不到实例页结果就直接判断“巡检没跑”

更稳的话法：

> Inspection 这类问题先分“任务定义、触发方式、结果查看、UTC 调度”四层；很多不是巡检能力坏了，而是把 `Inspection` 和 `InspectionJob`、实例页和任务页、本地时间和 UTC 混在了一起。

关联资料：
- [ACP Database & Middleware Inspection Center Quick Card](../notes/database-middleware-inspection-center-quick-card.md)

## 16. Data Services 实例升级管理问题，先怎么分 patch / major/minor？
先判断用户问的是哪种升级：
- **patch version upgrade**：更像走 unified-om 的 Instance Upgrade Management
- **major/minor version upgrade**：更像更新实例并选择受支持版本，不是都走同一套批量升级路径

先问什么：
- 当前组件与当前版本是什么
- 目标升级是 patch、minor 还是 major
- 问题发生在 Operator 列表、实例列表，还是批量升级执行阶段
- 当前 project / cluster 是否选对
- 如果切换过 project，cluster selector 是否已重新选择
- 现场关注的是“列表看不到实例”、“不能批量升级”，还是“升级顺序/时机和预期不一样”

不要先承诺什么：
- 不要把所有版本升级都承诺成可以走统一批量升级入口
- 不要把 project / cluster 切换后的列表空白，直接判成实例丢失
- 不要承诺多实例会严格按指定顺序升级

更稳的话法：

> 先区分 patch 和 major/minor。当前统一升级管理主打 patch version；如果是 major/minor，更像实例级受支持版本更新，不一定走同一套批量升级链路。另外切 project 后 cluster 选择器会重置，这类“列表没了”先别急着判故障。

关联资料：
- [ACP Database & Middleware Unified O&M Quick Card](../notes/database-middleware-unified-om-quick-card.md)

## 17. 这类问题更适合 FAQ 还是 Runbook？
适合 FAQ 的：
- 支持矩阵 / 协议 / 环境边界
- 配置改了为什么不生效、迁移前先确认什么
- Redis 客户端连接观测建议
- 某组件是否有页面入口 / 是否需要单独方案上架
- 某中间件实例内存高时先怎么分“正常占用 vs 真要扩容”
- Redis 单实例规格到底是在问“能不能配”还是“建不建议这样配”
- MySQL MGR 成员起不来时，先分实例问题还是集群脑裂/修复问题
- MySQL MGR binlog 告警时，先分复制异常、业务写入变化还是保留策略问题
- unified-om 下的备份 / 外部 S3 / inspection / patch 升级边界问法

适合 Runbook 的：
- 中间件连接异常
- MQ 创建失败 / 删除异常
- Nacos/数据库配置调整后不生效
- 中间件实例规格调整 / 扩容判断
- Redis 改规格后起不来 / namespace limit 冲突 / operator 收敛异常
- MySQL MGR 脑裂 / complete outage / 成员重加入失败
- MySQL MGR binlog 容量告警 / 参数与容量联动判断
- 扩展包上传 / 安装 / 审批 / 下错 cluster 这类执行链问题
