# TLS 在 Go 1.23+ 的重新签名代理后失败，出现 x509 负序列号

## 问题

在 Alauda 容器平台 4.3.x（base-v4.3.5 / acp-business-v4.3.6 ModuleConfigs）上，使用 Go 1.23 或更高版本构建的工作负载或平台组件可能无法与位于 TLS 拦截代理后面的端点建立 TLS 连接，出现错误 `tls: failed to parse certificate from server: x509: negative serial number`。TLS 拦截代理终止上游连接并重新签名，将其自己的 CA 签名证书替换到呈现给客户端的链中；当该替换证书携带负序列号时，Go 1.23+ TLS 客户端拒绝重新签名的证书并无法解析它。RFC 5280 §4.1.2.2 要求证书的 `serialNumber` 必须是非负整数，因此带有负序列号的证书不符合规范。

## 根本原因

Go 标准库证书解析器 `crypto/x509.ParseCertificate`（Go 1.23+）默认拒绝负序列号的证书，拒绝解析它们。使用 Go 1.22 或更早版本构建的组件不会拒绝负序列号的证书，因此在运行时升级之前，相同的代理呈现证书可以成功解析。因此，拒绝是标准库解析器中受 Go 版本限制的行为，而不是每个工具都强制执行的通用 X.509 规则：之前成功的连接仅在客户端构建在更严格的运行时上时开始失败，即使代理证书本身没有改变。

## 解决方案

用符合 RFC 5280 的证书替换 TLS 拦截代理呈现的证书，该证书的序列号为非负；正序列号的证书可以顺利解析，没有负整数编码，从而消除了解析失败。由于 RFC 5280 §4.1.2.2 强制要求 `serialNumber` 为非负，重新签发代理的重新签名叶证书并使用合规的正序列号可以恢复合规性，并让 Go 1.23+ 解析器接受该链。

## 诊断步骤

转储代理端点实际返回的证书链，以确认不受信任的自定义 CA 正在重新签名连接——这是 TLS 拦截的迹象：

```bash
openssl s_client -connect <host>:443 -showcerts
```

将 `<host>` 替换为正在诊断的代理端点——受影响的 Go 客户端试图通过拦截代理访问的主机名。

返回链中出现自定义 CA 而不是端点预期的颁发者，表明代理正在拦截并重新签名 TLS 会话。请注意，检查工具本身仍然显示这样的证书，包括携带负序列号的证书——拒绝特定于 Go 1.23+ 标准库解析器，因此失败是从受影响的 Go 客户端而不是从链转储中重现的。
