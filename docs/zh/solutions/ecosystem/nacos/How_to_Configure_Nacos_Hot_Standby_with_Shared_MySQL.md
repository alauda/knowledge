# 如何配置 Nacos 热备份以实现配置灾难恢复（共享 MySQL）

## 介绍

本指南描述了基于具备灾难恢复能力的 MySQL 的 **Nacos 2.5 配置数据** 的热备份灾难恢复（DR）拓扑。一个主 Nacos 集群（“Nacos A”）和一个备用 Nacos 集群（“Nacos B”）共享同一个逻辑 MySQL（通常是基于 MGR 的 DR 设置）。一个第三层负载均衡器位于两个集群前面，以便在站点故障时可以快速切换活动端点。

该计划适用于 ACP 3.18、4.0 和 4.1 上的 Nacos 2.5.x。

## 范围和限制

该计划仅涵盖 **配置资源热备份**。它不提供多写 — 两个 Nacos 集群同时写入同一数据库会导致数据损坏。

需要提前告知客户的其他限制：

1. **数据一致性**：由于两个集群写入同一逻辑数据库，高并发写入或复制延迟可能会导致活动和备用之间的瞬时不一致，短暂影响服务发现的准确性。该计划对配置工作负载容忍这种情况，但对命名工作负载则不然（见下文“临时服务不复制”）。
2. **资源争用**：双集群访问将负载集中在一个数据库上。大量写入流量可能会导致锁争用。
3. **切换复杂性**：从 A 切换到 B 时，残余的数据库负载或复制延迟可能会延长切换窗口。

> **风险**：该计划仅适用于 **配置数据的读侧热备份**。请勿在主动-主动写模式下使用 — 并发写入存在数据分歧的风险。

## Nacos 刷新工作原理

Nacos-server 针对 MySQL 有两条协调路径：

| 机制              | 默认间隔       | 设置                                                                                          |
| ---------------- | -------------- | --------------------------------------------------------------------------------------------- |
| 完整转储         | 6 小时         | `DUMP_ALL_INTERVAL_IN_MINUTE`（常量）                                                         |
| 增量转储         | 30 秒          | `dumpChangeWorkerInterval`（隐藏）受 `dumpChangeOn`（隐藏，默认 `true`）控制 — Nacos 2.5 |

实际上，直接写入底层数据库（带有匹配的时间戳更新）大约在 30 秒内传播到 Nacos-server 缓存，然后通过推送传播到客户端。

有关上游实现，请参见 Nacos 源代码中的 `console/src/main/java/com/alibaba/nacos/console/controller` 和 `config/src/main/java/com/alibaba/nacos/config/server/service/dump` 中的相关转储任务类。

## 架构

```text
                           ┌──────────────────────────┐
                           │    外部 LB / F5         │
        (active) ─────────►│  8848/http   9848/grpc   │◄───────── (standby)
                           └──────────────────────────┘
                            │                        │
                            ▼                        ▼
                    ┌───────────────┐        ┌───────────────┐
                    │    Nacos A    │        │    Nacos B    │
                    │   (primary)   │        │   (standby)   │
                    └───────┬───────┘        └───────┬───────┘
                            │                        │
                            └────────────┬───────────┘
                                         ▼
                       具备灾难恢复能力的 MySQL (MGR / 等效)
```

- Nacos A 是客户端访问的活动集群。
- Nacos B 读取相同的数据库并待命，准备接管。
- LB 有两个监听器 — `8848/http` 和 `9848/grpc` — gRPC 端口 **必须** 恰好为 `http + 1000`。

## 先决条件

1. 具备灾难恢复能力的 MySQL — Nacos 支持 MySQL 或任何兼容 MySQL 协议的存储。团队推荐的 DR 模式是内部 MGR 热备份方案（请查阅该数据库的 DR 操作手册）。
2. MySQL 端点必须可以从两个 Nacos 集群访问。
3. 用户/数据库（`nacos_config` 和 Nacos 用户）必须在共享逻辑 MySQL 上创建一次。
4. 在两个 Nacos 集群前面配置一个外部负载均衡器或 F5 设备。
5. 两个 Nacos 集群可以使用 [如何部署 Nacos 2.5](./How_to_Deploy_Nacos_2.5.md) 进行安装。本文档仅说明独立安装与 DR 安装之间的差异。

## 操作步骤

### 1. 配置 DR MySQL

根据团队的 MySQL DR 操作手册（MGR 热备份或等效）部署 DR MySQL。仅需 **初始化 Nacos 架构一次** — Nacos B 将重用 Nacos A 的表。

### 2. 部署 Nacos A（主）

按照 [如何部署 Nacos 2.5](./How_to_Deploy_Nacos_2.5.md) 的步骤进行。当配置 `db.host` 时，指向从集群 A 可访问的 DR MySQL 端点。

### 3. 部署 Nacos B（备用）

按照相同的计划进行部署，需注意两个差异：

1. `db.host` 必须指向从集群 B 可访问的 DR MySQL 端点（通常是不同的读/写路由，但最终是相同的逻辑数据库）。
2. **JWT 签名密钥必须与 Nacos A 匹配**。登录 A 的用户会收到用 A 的密钥签名的令牌；一旦 LB 切换到 B，B 必须接受该令牌，因此两个集群需要相同的密钥。

> **注意**：您不需要将 Nacos 管理用户复制过来 — 它存在于共享 MySQL 的 `users` 表中，因此 B 会自动继承。`Server Identity Key` / `Server Identity Value` 是用于同一集群 Nacos pod 之间的 **集群内对等身份验证** 头；它们在 A 和 B 之间可以不同，而不会破坏 DR 计划。

### 4. 配置外部负载均衡器

在 LB 上配置两个监听器：

| 端口   | 协议     | 目的                                                                      |
| ------ | -------- | ------------------------------------------------------------------------- |
| `8848` | HTTP     | 初始连接/握手。                                                          |
| `9848` | gRPC     | Nacos 与客户端之间的实时推送。必须恰好为 `8848 + 1000`。               |

两个监听器最初指向 Nacos A。

## 故障转移

### 复制延迟预算

由备用回答的客户端请求的总传播延迟有两个组成部分：

1. 数据库复制延迟 — 由底层 MySQL DR 机制决定。
2. Nacos 缓存刷新 — 受 30 秒增量转储周期的限制（带有一些数据库扫描的变化）。

因此，Nacos DR RTO 至少为 **`database DR RTO + 30 seconds`**，在持续的数据库负载下或长时间的完整转储过程运行时可能会超过该值。

### 故障转移步骤

1. **验证备用数据完整性** — 登录 Nacos B 的监控面板，确认预期的配置存在且是最新的。如果 MySQL 复制存在间隙，请立即显示这些间隙，以便操作员了解可能缺失的内容。
2. **将 LB 切换到 Nacos B** — 同时切换 `8848/http` 和 `9848/grpc` 两个监听器。端点不匹配（HTTP 指向 A，gRPC 指向 B）将破坏推送语义。

## 验证

以下验证场景假设 LB 仍指向 Nacos A。

### 配置同步 — 创建

1. 在 Nacos A 上创建配置 `test.yaml`，值为 `a: 1`，`b: 2`。
2. 打开 Nacos B 的监控面板，验证 `test.yaml` 在 ~30 秒内可见且数据相同。

### 配置同步 — 更新

1. 在 Nacos A 上，将 `a` 更改为 `111`。
2. 刷新 Nacos B 的监控面板，确认 `a` 现在为 `111`。

### 配置同步 — 删除

1. 在 Nacos A 上删除 `test.yaml`。
2. 确认它已从 Nacos B 中消失。

### 命名数据行为

命名数据沿着临时/持久边界进行拆分：

- **临时实例**（基于心跳驱动，默认适用于大多数 Spring Cloud / Dubbo 应用）仅存在于每个 Nacos 集群的内存中。它们 **不** 复制。
- **持久实例**（`ephemeral=false`）存储在 MySQL 的 `instances` 风格表中，因此 *确实* 会出现在备用中 — 但实时健康状态、推送订阅和调度器状态仍在内存中，因此持久注册无法通过此 DR 方案无缝提供服务。

运行以下操作以确认临时命名流量 **不** 是此计划的一部分：

1. 在 Nacos A 上注册一个临时实例 — 它不应出现在 Nacos B 中。
2. 在 Nacos B 上注册一个临时实例 — 它不应出现在 Nacos A 中。

对于需要命名数据 DR 覆盖的工作负载，将其视为单独的设计，**不要** 依赖此仅配置的计划。
