# RabbitMQ 最佳实践

:::info 适用版本
ACP 3.14 及更高版本。大多数规模和部署指导也适用于早期基于操作员的 RabbitMQ 部署。
:::

## 介绍

RabbitMQ 广泛用于应用解耦、异步处理、事件分发和请求-响应工作负载。在 Kubernetes 环境中，生产稳定性取决于正确的规模、队列设计、存储选择和部署隔离。

在规划新的 RabbitMQ 集群或审查现有集群时，请将本指南作为基准。

## 核心概念

| 术语     | 描述                                                  |
| -------- | ----------------------------------------------------- |
| Producer | 发布消息的客户端。                                   |
| Exchange | 接收消息并根据绑定进行路由。                         |
| Queue    | 存储消费者的消息。                                   |
| Binding  | 交换机与队列之间的路由规则。                         |
| Consumer | 从队列中读取消息的客户端。                           |
| Broker   | 一个 RabbitMQ 节点。一个集群包含一个或多个代理。   |

## 交换机类型

| 类型      | 用例                                                       |
| --------- | ---------------------------------------------------------- |
| `direct`  | 精确路由键匹配。适合点对点路由。                         |
| `fanout`  | 广播到所有绑定的队列。                                   |
| `topic`   | 基于模式的路由。适合选择性发布/订阅。                   |
| `headers` | 根据消息头而非路由键进行路由。                           |

当业务需要灵活的路由增长时，优先使用 `topic` 交换机。仅在意图明确的广泛广播语义时使用 `fanout`。

## 容量规划

### 内存

RabbitMQ 的内存使用受消息大小、积压大小、持久性、消费者速度和队列类型的影响。较大的积压会增加代理的内存压力和磁盘使用。

建议：

- 尽可能保持队列积压较低。
- 监控内存警报和队列增长。
- 除非资源隔离得到验证，否则避免在同一节点上共置内存密集型业务工作负载。

### CPU

CPU 使用主要来自路由、协议处理、队列管理、TLS 和插件开销。

建议：

- 根据峰值发布和消费速率而非平均负载来配置 CPU。
- 分别基准测试法定队列和镜像工作负载，而不是简单的经典队列。
- 当多个大型 RabbitMQ 实例共享同一节点时，检查 Erlang 调度程序绑定行为。

### 磁盘

持久队列需要能够吸收正常写入吞吐量和突发积压的存储。

建议：

- 对于生产集群，使用稳定的持久卷。
- 优先选择具有可预测延迟的存储类。
- 根据消息速率、保留时间、耐久性模式和灾难恢复复制开销来配置容量。
- 保持操作余量以应对故障转移、重新排队和临时积压峰值。

## 部署建议

### 操作员和实例布局

- 对于生产集群，使用多个副本。
- 使用 Pod 反亲和性将副本分布在节点之间。
- 在需要灾难恢复或迁移时，保持集群之间的插件集一致。
- 除非有明确理由不同，否则在各个环境中使用相同的服务暴露模型。

### 服务类型

根据访问模型选择服务暴露：

- 对于集群内应用，使用 `ClusterIP`。
- 当需要外部访问且平台网络在节点级别管理时，使用 `NodePort`。
- 当可用 MetalLB 或云负载均衡器时，使用 `LoadBalancer`。

### 持久性

除非设计上可以接受数据丢失，否则不要在没有持久性的情况下运行生产消息工作负载。

### 额外配置

在创建时检查以下项目：

- 副本数量
- 存储类和存储大小
- CPU 和内存请求及限制
- 服务类型
- 额外插件
- 亲和性、反亲和性和容忍度
- 环境配置，如调度程序绑定或插件路径

## 集成指导

- 优先通过稳定的服务端点访问，而不是单个 Pod IP。
- 将交换机、队列和绑定的创建纳入版本控制的应用或平台自动化中。
- 将客户端重试行为与代理故障转移行为对齐。
- 设计消费者以容忍重复交付。

## 操作指导

- 跟踪队列深度、消费者数量、消息速率、内存警报和磁盘警报。
- 及早调查增加的积压。慢消费者通常在资源耗尽之前变得可见。
- 首先在非生产环境中验证插件更改、灾难恢复设置和迁移程序。

## 常见风险

- 队列中积累过多消息
- 持久工作负载的存储不足
- 多副本集群缺少反亲和性
- 插件增长失控
- 将 RabbitMQ 视为持久的长期存储而非消息系统

## 参考建议

- 当集群必须满足严格的延迟或吞吐量目标时，使用专用节点。
- 优先选择显式路由和队列生命周期管理，而不是临时控制台驱动的更改。
- 在更改队列类型、插件或服务暴露模式后重新运行容量测试。
