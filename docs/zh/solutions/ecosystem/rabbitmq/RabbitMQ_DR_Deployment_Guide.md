# RabbitMQ 灾难恢复部署指南

:::info 适用版本
ACP 3.14, 3.15 和 3.16。
:::

## 介绍

本指南涵盖基于 Shovel 的热备份 RabbitMQ 灾难恢复解决方案的部署。源集群处理生产流量，目标集群保持复制消息，仅在故障转移后变为活动状态。

## 灾难恢复特性

| 项目                 | 值                                                  |
| -------------------- | ---------------------------------------------------- |
| 复制引擎           | RabbitMQ Shovel                                      |
| 模式                 | 接近实时的异步复制                                  |
| 推荐拓扑           | 源交换机到目标交换机                                |
| RTO                  | 几分钟，具体取决于手动故障转移                      |
| RPO                  | 几秒到几分钟，具体取决于堆积量和网络                |

## 风险和限制

- Shovel 是单进程复制，需要操作监控。
- 当网络或目标性能不足时，复制延迟会增加。
- 在不稳定期间，源和目标消息状态可能会出现偏差。
- 消费者必须容忍故障转移后的重复消息。

## 前提条件

- 源和目标 RabbitMQ 集群在不同的故障域中创建。
- 站点之间的网络连接稳定。
- 目标集群存储已根据完整的复制堆积量进行调整。
- 源和目标交换机、队列和绑定需提前规划。
- 在一侧启用 Shovel 插件，最好是在目标集群上。

## 启用插件

```yaml
spec:
  rabbitmq:
    additionalPlugins:
      - rabbitmq_shovel
      - rabbitmq_shovel_management
```

## 推荐的部署模式

### 源交换机到目标交换机

当目标集群应保留交换机语义时使用此模式。Shovel 在源侧创建一个内部队列，并将消息重新发布到目标交换机。

当源交换机是 `topic` 交换机时，路由键 `#` 可以保护该交换机的所有路由键。

### 源交换机到目标队列

仅在目标侧故意终止到特定队列时使用此模式。RabbitMQ 在目标侧使用默认交换机将消息路由到该队列。

## 创建源和目标对象

在配置 Shovel 之前：

1. 在两个集群上创建所需的交换机。
2. 创建目标队列。
3. 创建目标绑定。
4. 确认源路由键和目标路由规则一致。

Shovel 使用的自动生成的 `amq.*` 内部队列不需要手动创建。

## 配置 Shovel

在 RabbitMQ 管理中配置：

1. 源 URI 和源对象
2. 目标 URI 和目标对象
3. 重连延迟
4. 确认模式

推荐设置：

- `重连延迟`：非零值，例如 `5`
- `确认模式`：`on confirm`

## 部署验证

1. 确认 Shovel 状态变为 `running`。
2. 确认内部队列在源侧正确创建并绑定。
3. 向源交换机发布测试消息。
4. 确认目标队列接收到预期的消息。
5. 让测试运行足够长的时间，以观察在持续发布负载下的堆积行为。

## 容量考虑

- 目标集群保持复制消息而不进行正常的消费者排空。
- 队列保留和消息过期应明确规划。
- 站点之间的带宽必须超过峰值复制需求。
- 目标集群的磁盘、CPU 和内存配置应与源工作负载配置相匹配。
