# RabbitMQ 热备份灾难恢复与 Shovel

:::info 适用版本
ACP 3.14、3.15 和 3.16。
:::

## 介绍

该解决方案在不同站点构建两个 RabbitMQ 集群。源集群处理生产流量。目标集群接收复制的消息，仅在故障转移决策后使用。

复制通过 RabbitMQ Shovel 实现。Shovel 作为客户端，从源侧读取消息并将其发布到目标侧。

## 架构

RabbitMQ 通过交换机和队列路由消息。Shovel 可以从该模型的任一侧消费并发布到目标模型的任一侧。对于热备份，实际选择为：

- 源交换机到目标交换机
- 源交换机到目标队列

推荐的模式是源交换机到目标交换机。使用 `topic` 交换机和路由键 `#`，一个 Shovel 可以覆盖该交换机的所有匹配流量。

## 限制

- Shovel 不是一个完整的主动-主动复制引擎。
- 复制是异步的，因此在故障期间仍可能会丢失一些数据。
- 目标集群存储复制的消息而不进行正常消费，因此存储必须根据保留的积压数据进行调整。
- 故障转移后可能会发生重复消费，因为已消费的源消息可能仍存在于目标侧。
- 客户端生产者和消费者端点不会自动切换。
- 在不稳定的网络条件下，Shovel 可能会删除其自动创建的内部队列，并可能丢失在途数据。

## 推荐拓扑

| 项目            | 推荐                                                         |
| --------------- | ------------------------------------------------------------ |
| Shovel 位置     | 仅在一侧启用 Shovel 插件，通常是目标集群                   |
| 源模式         | 交换机                                                       |
| 目标模式       | 交换机                                                       |
| 交换机类型     | 当需要广泛的交换级保护时使用 `topic`                       |
| 路由键         | 使用 `#` 以实现完整的交换覆盖                               |

## 启用 Shovel 插件

将以下插件添加到 `RabbitmqCluster`：

```yaml
spec:
  rabbitmq:
    additionalPlugins:
      - rabbitmq_shovel
      - rabbitmq_shovel_management
```

## 关键 Shovel 参数

| 参数                   | 描述                                                  |
| ---------------------- | ----------------------------------------------------- |
| `Name`                 | 在管理 UI 中显示的 Shovel 名称                       |
| `Source`               | 源协议、URI 以及交换机或队列设置                     |
| `Destination`          | 目标协议、URI 以及交换机或队列设置                   |
| `Reconnect delay`      | 链接故障后重新连接的延迟                             |
| `Acknowledgement mode` | `no ack`、`on publish` 或 `on confirm`               |

在消息持久性比吞吐量更重要时，优先使用 `on confirm`。

## 部署流程

1. 创建源集群和目标集群。
2. 在选择的集群上启用 Shovel 插件。
3. 在两侧创建所需的交换机、队列和绑定。
4. 在 RabbitMQ 管理中配置 Shovel。
5. 验证自动创建的内部队列是否出现在源侧。
6. 向源交换机发布测试数据。
7. 确认目标队列接收到复制的消息。

## 监控

Shovel 的健康状况可以通过 Shovel 状态和队列行为来判断：

- Shovel 状态应从 `starting` 转变为 `running`。
- 源侧的内部队列通常显示为 `amq.gen-*`。
- 如果排队的消息持续增长，目标侧的交付速度过慢。
- 如果排队的消息保持在零以上且没有变化，复制可能已停滞。
- 如果消费者确认率远低于发布率，复制吞吐量不足。

## 重要考虑事项

- 为完整的备用保留调整目标存储大小。
- 在调用解决方案准备就绪之前，使用业务客户端测试故障转移。
- 在消费者中构建幂等性或重复处理。
- 将此模式用于热备份，而不是多主消息传递。
