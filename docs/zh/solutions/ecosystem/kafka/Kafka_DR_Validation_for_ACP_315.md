# Kafka 灾难恢复验证 for ACP 3.15

:::info 适用版本
ACP 3.15.
:::

## 介绍

在使用 MirrorMaker 2 部署 Kafka 热备份灾难恢复后，验证主题复制、数据复制、消费者组同步、监控和故障转移行为。

## 前提条件

- 源 Kafka 集群、目标 Kafka 集群和 MirrorMaker 2 实例已部署。
- MirrorMaker 2 自定义资源已准备好。
- 如果需要告警和监控面板验证，则已配置监控。
- 您有可用的测试主题和测试生产者/消费者工具。

## 验证主题同步

1. 在源集群上创建一个主题。
2. 检查该主题是否出现在目标集群上。

新主题默认不会立即同步。MirrorMaker 2 根据 `refresh.topics.interval.seconds` 发现新主题，该值通常设置为 600 秒。如果需要更快的发现，可以降低此值进行测试。

## 验证数据同步

1. 向源主题生产测试记录。
2. 从目标集群的相应主题消费。
3. 确认目标主题接收到预期的记录。

示例生产者和消费者命令：

```bash
kafka-console-producer.sh \
  --bootstrap-server <source-bootstrap>:9092 \
  --topic my-topic

kafka-console-consumer.sh \
  --bootstrap-server <target-bootstrap>:9092 \
  --topic my-topic \
  --from-beginning
```

## 验证消费者组同步

1. 向源主题生产记录。
2. 使用固定组（例如 `group1`）在源集群上消费部分记录。
3. 停止源消费者。
4. 继续向源主题生产记录。
5. 检查目标集群以确认同步的消费者组。
6. 在目标集群上启动相同组的消费者，并确认它从同步的偏移量继续，而不是总是从头开始。

消费者组偏移量是定期同步的，而不是实时的。如果源集群在最新偏移量检查点到达目标集群之前失败，可能会发生一些重复消费。

## 验证监控

1. 向源主题生产流量。
2. 打开 MirrorMaker 2 监控面板。
3. 确认复制的主题和流量出现在面板中。
4. 检查堆积量、复制延迟、记录年龄、CPU、JVM 内存和任务数量。

## 验证故障转移

1. 在源集群上使用固定消费者组生产和消费记录。
2. 在消费部分记录后停止源消费者。
3. 继续生产更多记录。
4. 通过在受控测试环境中停止 Kafka operator 或 broker 工作负载来模拟源集群故障。
5. 将应用程序引导地址切换到目标集群。
6. 在目标集群上生产和消费。
7. 确认消费者组可以从同步的偏移量继续。
8. 恢复源集群，并单独测试计划的切换回程序。

## 重要注意事项

- 仅在非生产或批准的灾难恢复演练窗口中运行故障转移验证。
- 在某些故障时序中，预期会发生重复消费。应用程序必须是幂等的，或以其他方式容忍重复消息。
- 主题创建和消费者组同步使用不同的刷新间隔。
- 记录演练中观察到的 RTO 和 RPO，而不是仅依赖理论值。
