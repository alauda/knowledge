# Redis 最佳实践

:::info 适用版本
Alauda Container Platform **v3.18+**（在此更新时支持的 operator）。下面的架构建议和规模指导最初在 v3.12–v3.14 上进行了验证，并且在当前版本中仍然广泛适用。当特定数字（参数模板名称、容器限制、基准数字）与当前 operator 默认值偏离时，优先使用您已安装的 operator 提供的值。
:::

## 介绍

Redis 是一个高性能的内存数据存储，广泛用作缓存、消息代理、排行榜等。在 Kubernetes 上容器化 Redis 提供了一致、可移植和可重现的运行时，但也带来了关于部署、配置、持久性、性能调优、实例生命周期管理和应用集成的新挑战。

本文档收集了我们推荐的在 Kubernetes 上运行 Alauda Cache Service for Redis OSS 的最佳实践。遵循这些实践可以帮助您实现持久、高可用和高性能的 Redis 实例，同时保持可管理的操作开销。

## 架构概述

Alauda Cache Service for Redis OSS 提供了两种托管架构以适应不同的工作负载。

### 哨兵模式 (Redis Sentinel)

哨兵模式是基于 Redis 主从复制构建的高可用解决方案。Redis Sentinel 监控实例状态，并在主节点故障时自动将从节点提升为主节点。

主要特征：

- **易于操作。** 部署和管理比集群模式更简单。
- **高可用。** 自动故障转移在节点故障时保持服务可用。
- **扩展有限。** 仅支持垂直扩展和通过从节点的读取扩展。没有原生的水平分片。
- **哨兵可用性。** 哨兵进程本身必须保持高可用（至少运行三个哨兵 pod）。

### 集群模式 (Redis Cluster)

集群模式是 Redis 的原生水平可扩展架构。数据通过哈希槽算法在多个节点之间分片，使数据集的大小远远超出任何单个节点的内存。

主要特征：

- **高可用。** 从节点可以自动接管故障的主节点。
- **水平扩展。** 可以在线添加或删除分片（及其托管的节点）。
- **负载均衡。** 读取和写入在主节点之间分配。
- **分布式数据集。** 避免大数据集集中在单个节点上造成的热点。
- **更高的复杂性。** 分片、哈希槽、数据迁移和重新平衡增加了操作复杂性。

### 架构选择矩阵

|               | 需要无限水平扩展 | 单节点或小占用 | 大数据集 (>8 GB) | 高可用性 |
| ------------- | :---------------: | :------------: | :---------------: | :------: |
| 集群模式      |        是         |       否       |         是        |   是     |
| 哨兵模式      |        否         |       是       |         否        |   是     |

## 资源规模

### 内存

Redis 定期将内存数据快照保存到磁盘。此快照模型速度快，但在快照之间写入的数据可能会在 Redis 被终止时丢失。

对于生产部署，我们建议将每个 Redis 分片的内存保持在 **8 GB** 以下。这个建议有两个原因：

1. **单线程事件循环。** Redis 在一个线程中处理网络 I/O 和数据操作。随着数据集的增长，响应时间可能会增加，最终导致客户端可见的延迟或停顿。
2. **持久性和备份开销。** 较大的数据集使 RDB/AOF 快照、复制和备份/恢复操作变得更慢且更昂贵。

如果单个分片超过 8 GB，请考虑使用集群模式对数据进行分片，而不是进一步扩大分片。

### CPU

在 Kubernetes 中，您可以选择使用 kubelet 的静态 CPU 管理器来专用 CPU 核心：

```yaml
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
cpuManagerPolicy: static
```

更新 kubelet 配置后，重启 kubelet：

```bash
systemctl restart kubelet
```

使用 `cpuManagerPolicy: static`，请求整数 CPU 值的 pod 将获得独占核心，这对 CPU 敏感的 Redis 工作负载非常有帮助。请选择性使用——过度分配独占 CPU 会浪费容量。

#### 单线程与多线程规模

Redis 在命令执行上本质上是单线程的。在持久化期间，Redis 使用 `fork()` 创建一个子进程来写入 RDB 文件；父进程继续在其单线程上处理请求。将每个分片的 CPU 限制设置为 **2 核心** 可确保前台线程在持久化期间继续处理流量。

如果您启用 Redis I/O 线程（Redis 6.0+），I/O 可以在额外的核心上并行化。请记住，只有某些操作（网络 I/O、某些 Lua 路径）能从线程中受益——大多数命令执行仍然是单线程的。将 CPU 规模设置为超过 2 核心仅对具有大量 I/O 并发的工作负载有帮助。

#### 参考基准 — 单线程

以下数字是 **历史参考数字**，在 Redis 6.0 和 operator v3.14 上捕获。它们说明了单线程与 I/O 线程扩展的形状；在当前硬件和 Redis 7.x 上的绝对吞吐量将更高。**在规模化之前，请针对您的实际实例重新运行 `redis-benchmark`。**

没有 I/O 线程的 Redis 实例，使用 `redis-benchmark` 进行测试（100 个并发连接，每个连接 100,000 个请求，50% SET / 50% GET）：

| CPU 核心 | 吞吐量 (ops/sec) |
| -------- | ----------------- |
| 1        | 47,760            |
| 2        | 50,143            |
| 4        | 51,443            |
| 8        | 50,440            |

超过 2 核心几乎没有改进。

#### 参考基准 — 启用 I/O 线程

相同工作负载，4 vCPU / 8 GB 实例，变化 I/O 线程数量：

| 线程数 | 吞吐量 (ops/sec) |
| ------- | ----------------- |
| 1       | 51,847            |
| 2       | 77,374            |
| 4       | 115,934           |
| 8       | 138,089           |

I/O 线程可以显著提高吞吐量，但实际收益取决于工作负载、负载大小和网络。在依赖绝对值之前，请使用您自己的基准进行验证。

:::tip
默认情况下，将每个分片的 CPU 固定为 2 核心。仅在您已启用 I/O 线程并确认收益后才增加它。
:::

### 磁盘

默认情况下，operator 将持久卷大小设置为 `maxmemory` 的两倍。这是一个合理的起点，但并不保证对每个工作负载都有足够的空间。RDB 和 AOF 文件的大小取决于实际内存使用情况和写入模式，而不是 `maxmemory` 上限。

如果内存利用率很高，持久性文件会相应增长。监控您的磁盘使用情况，并在磁盘压力导致故障之前增加 PVC 大小或调整数据模型（TTL，更高效的编码）。

#### RDB 加载和保存参考

历史参考数据（Redis 6.0，operator v3.14）——在 HDD 上测量，顺序读取 1117 MiB/s 和写入 772 MiB/s。使用表格估算数据集大小作为加载/保存时间的 **形状**。SSD 存储和 Redis 7.x 都显著减少了这些时间；在依赖绝对值之前，请在您自己的存储上进行基准测试：

| RDB 大小                     | 加载时间 | 保存时间 |
| ---------------------------- | -------- | -------- |
| 7.6 GB (8.77 GB 使用内存)    | 52.83 s  | 59.77 s  |
| 6.7 GB (7.10 GB 使用内存)    | 27.66 s  | 52.72 s  |
| 5.7 GB                       | 24.05 s  | 43.09 s  |
| 4.8 GB                       | 18.61 s  | 36.84 s  |
| 3.8 GB                       | 14.51 s  | 29.86 s  |
| 2.9 GB                       | 10.78 s  | 22.24 s  |

使用这些数字来设置您的 pod 启动探针延迟和 `preStop` 宽限期。例如，对于约 7.6 GB 的 RDB，至少允许 60 秒让 pod 加载，另加 60 秒在强制终止之前，以便 Redis 可以干净地持久化其数据集。随着数据集的增长，按比例增加这些值。

## 部署

### Operator 拓扑

Redis Operator 可以以两种模式部署：

- **集群模式。** 单个 operator 管理任何命名空间中的实例。
- **命名空间模式。** Operator 实例被限制在特定命名空间内，适用于强租户隔离。

选择与您的租户模型匹配的模式。

### 创建实例

#### 调度和反亲和性

集群模式支持三种推荐的反亲和性策略：

- **全 pod 硬反亲和性。** 每个 Redis pod 必须运行在不同的节点上。提供最强的故障隔离，但需要的节点数量至少与 pod 数量相等。当节点不足时，部署会失败。
- **分片内硬反亲和性。** 同一分片的主节点和从节点必须运行在不同的节点上；来自不同分片的 pod 可以共存。保护单节点故障，同时需要更少的节点。
- **分片内软反亲和性。** 主节点和从节点应尽可能运行在不同的节点上，但在资源紧张时可以共存。在牺牲隔离性的情况下最大化可部署性。

| 策略                     | 描述                                                                         | 优点                             | 缺点                                   | 单一故障自愈               | 节点故障时的数据完整性      |
| ------------------------ | -------------------------------------------------------------------------- | -------------------------------- | -------------------------------------- | -------------------------- | --------------------------- |
| 全 pod 硬反亲和性        | 所有 pod 必须在不同的节点上                                               | 最佳的高可用性和负载均衡       | 当节点稀缺时，部署会失败              | 有保障                     | 当故障节点 < 副本时完好     |
| 分片内硬反亲和性        | 同一分片的主节点和从节点必须在不同的节点上                             | 每个分片的强故障隔离           | 当节点稀缺时，部署会失败              | 可能                       | 当故障节点 < 副本时完好     |
| 分片内软反亲和性        | 主节点和从节点应在不同的节点上；如有需要可以共存                       | 适应有限的容量                 | 当共存时隔离性较弱                    | 可能                       | 可能完好                     |

在哨兵模式下，默认调度策略是 **分片内软反亲和性**——主节点和从节点在可能的情况下分布在不同的节点上，但可以共存以允许在较小集群上进行部署。对于生产哨兵工作负载，如果您的节点容量允许，建议使用硬反亲和性。

## 应用集成

### 访问模式

选择与您的拓扑匹配的连接模式。

| 架构        | 访问模式                | 推荐端点                          | 备注                                                                                                                                                                                  |
| ----------- | ----------------------- | --------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 哨兵        | 集群内访问              | 哨兵地址                          | 客户端连接到哨兵地址（默认端口 `26379`），并通过哨兵发现数据节点。                                                                                                               |
| 哨兵        | 外部访问                | 数据节点上的 NodePort + 哨兵上的 NodePort | 哨兵仅处理发现，因此数据节点的端口也必须暴露。客户端连接到哨兵 NodePort。                                                                                                         |
| 集群        | 集群内访问              | 无头服务（每个节点）              | 客户端使用服务 DNS 发现所有集群节点（默认端口 `6379`）。连接可能解析到任何 pod，因此客户端必须是集群感知的，以跟随 `MOVED` 重定向。                                             |
| 集群        | 外部访问（无代理）      | 每个数据节点上的 NodePort        | 客户端直接连接到数据节点，使用多个 NodePorts。提供了没有代理的弹性，但客户端必须是集群感知的，并配置多个端点。                                                                  |

### 客户端配置示例

我们内部测试中使用的参考客户端是 `jedis` Java 客户端：

```xml
<dependency>
  <groupId>redis.clients</groupId>
  <artifactId>jedis</artifactId>
  <version>3.7.0</version>
</dependency>
```

连接到 Redis 集群时，请在集群模式下配置客户端，并提供所有集群端点——例如使用 Spring Boot：

```yaml
spring:
  redis:
    cluster:
      nodes: <node-1-host>:<port>,<node-2-host>:<port>,<node-3-host>:<port>
```

连接到集群实例的非集群感知客户端会失败，并出现 `ERR SELECT is not allowed in cluster mode` 或 `MOVED <slot> <host:port>` 等错误。有关集群模式连接错误的故障排除，请参见故障排除文章。

## 重要考虑事项

- **将分片内存限制在 8 GB。** 超过此点，Redis 尾部延迟会降低，持久化操作变得昂贵。
- **根据持久性需求匹配参数模板。** 使用 RDB 作为默认；仅在无法容忍任何数据丢失时切换到 AOF；当 Redis 纯粹作为缓存时使用无持久化。
- **根据实际使用情况规划磁盘，而不是 `maxmemory`。** `maxmemory` 的两倍是起点；监控实际消耗。
- **根据节点容量选择反亲和性。** 硬反亲和性是更安全的默认选项，但需要足够的节点；当容量是限制条件时，软反亲和性是可以接受的。
- **集群模式需要集群感知客户端。** 在上线之前确认客户端和连接字符串，以避免 `MOVED` 和 `SELECT` 错误。
- **对于哨兵模式客户端，使用哨兵地址，而不是数据节点地址。** 否则故障转移将不透明。
