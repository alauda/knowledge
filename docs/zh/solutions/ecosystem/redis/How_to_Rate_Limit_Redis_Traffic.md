# 如何限制 Redis 流量

## 介绍

本指南解释了控制 Redis 流量的策略，以防止过度负载可能影响集群稳定性。它涵盖了基于带宽的速率限制与基于连接的节流之间的权衡，并提供了在生产环境中管理 Redis 流量的实用建议。

## 背景

### 什么导致 Redis 流量过大？

Redis 流量过大通常是由以下一个或多个原因造成的：

1. 应用程序的高请求量
2. 大键的存储和频繁访问
3. 使用返回大量数据的危险命令

### 为什么直接的带宽节流是有问题的

:::warning
直接对 Redis Pod 应用带宽限制可能导致级联故障。
:::

如果对 Redis 应用带宽限制，流量激增将饱和可用带宽，导致：

- 客户端请求在 Redis 操作上阻塞。当 API 并发超过 10,000 时，所有 10,000 个 TCP 连接（默认的 `maxclients`）争夺有限的带宽，导致响应缓慢。
- 广泛的 TCP 请求超时，这会传播为上游 API 超时。
- Redis 主从复制和集群心跳同步中断，增加复制延迟，削弱高可用性。

因此，**不推荐使用基于带宽的速率限制**。请改用基于连接的限制和上游 API 节流。

## 节流前的考虑

在大多数生产工作负载中，Redis 本身很少成为瓶颈：

- Sentinel 模式可以维持约 500k QPS，使用读写分离时可达约 1M QPS。
- 对于更高的吞吐量，请使用集群模式。

在 Redis 层添加流量控制之前，请评估：

- 上游 API 层是否能够承受流量
- 下游数据库是否能够承受流量
- 是否应在网关或 API 层应用节流

当带宽不是瓶颈时，即使在高并发下，Redis 也能迅速响应。然而，大键的存在可能会不成比例地消耗带宽。限制最大客户端连接通常比节流带宽更有效：当流量激增超过连接限制时，客户端会立即收到错误，而不是在缓慢的 Redis 上排队。

## 操作步骤

### 方法 1：限制最大客户端连接 (`maxclients`)

在您的 Redis 实例配置中设置 `maxclients` 参数，以控制连接级别的并发。

- **默认值**：`10000`
- **推荐调整**：根据实际并发需求减少到 `5000` 或更低。

通过 Redis 管理 UI 中的实例参数配置或通过 YAML 的 `customConfig` 字段更新 `maxclients`：

```yaml
spec:
  customConfig:
    maxclients: "5000"
```

:::tip
基于连接的限制优于基于带宽的限制，因为过多的客户端会快速失败并显示明确的错误，而不是经历缓慢的超时。
:::

### 方法 2：在 API 或网关层应用速率限制

在离客户端更近的地方实施节流以保护 Redis：

1. **基于 API 网关的节流**：
   - 使用 Kgateway 或 Kong 等 API 网关根据路由、消费者或 API 密钥应用速率限制规则。
2. **应用级节流**：
   - 在您的业务 API 中直接实现速率限制（例如，令牌桶或漏桶算法）。

### 方法 3：检测和修复大键

大键在访问时会消耗不成比例的带宽和 CPU。识别并修复它们：

1. 使用您的 Redis 平台提供的大键检测工具。
2. 重构应用逻辑，将大键拆分为更小的结构。
3. 避免频繁完全读取大型哈希、列表或集合。

### 方法 4：减少使用危险命令

某些命令在单次调用中返回大量数据，占用大量带宽并阻塞 Redis。避免或限制以下命令：

- `KEYS`
- `HGETALL`
- `SMEMBERS`

在可能的情况下，用迭代替代方案替换这些命令：

- 使用 `SCAN` 替代 `KEYS`
- 使用 `HSCAN` 替代 `HGETALL`
- 使用 `SSCAN` 替代 `SMEMBERS`

您还可以在 Redis 实例级别禁用危险命令。请参阅相关解决方案 **如何管理危险的 Redis 命令**。

## 重要考虑事项

- 在 Pod 级别直接进行带宽节流可能导致广泛的服务降级。避免这种方法。
- 在多个层面应用速率限制最为有效：网关、API 和 Redis 连接限制。
- 定期监控大键。频繁访问的单个大键即使在连接数较低时也能饱和 Redis 带宽。
- 调整 `maxclients` 时，确保该值仍能容纳合法的峰值流量；设置过低可能导致正常操作期间的连接拒绝。
