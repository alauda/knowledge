# Handle VPC Egress Gateway BFD session issues

This article provides solutions for two VPC Egress Gateway (VEG) BFD session issues. Select the scenario that matches the observed behavior.

## Scenario 1: BFD sessions repeatedly transition between Up and Down

**Applicable versions:** ACP v4.3.0 and v4.3.1

### Symptoms

After BFD is enabled for the VEG, `bfdd-control status` reports one or more sessions that repeatedly transition between `Up` and `Down`. Application outbound traffic can be interrupted when OVN marks the corresponding VEG next hop as unavailable.

### Causes

BFD packets must be processed within the negotiated detection time. The following conditions can cause an established session to time out repeatedly:

- In ACP v4.3.0 and v4.3.1, the default CPU request and limit for the VEG `bfdd` container are both `50m`. CPU throttling or node scheduling delays can prevent `bfdd` from processing BFD packets in time.
- When the VEG uses `minRX: 100` and `minTX: 100`, the session is more sensitive to brief scheduling delays and packet loss.

These conditions can occur independently or together.

### Resolution

Upgrade to ACP v4.3.2 or later. The default CPU request and limit for the VEG `bfdd` container are both `200m` in these versions.

If you cannot upgrade ACP, apply the following changes to the affected VEG.

:::warning
Changing `spec.resources` or `spec.bfd` rolls out the VEG Deployment and recreates its Pods. For a single-replica VEG, schedule a maintenance window because application outbound traffic can be interrupted during the rollout.
:::

Edit the affected VEG:

```bash
kubectl -n <namespace> edit veg <veg-name>
```

Set the CPU request and limit to `200m`:

```yaml
spec:
  resources:
    requests:
      cpu: 200m
    limits:
      cpu: 200m
```

If the session remains unstable after increasing the CPU, and the VEG explicitly uses a 100 ms detection interval, set `minRX` and `minTX` to `1000`:

```yaml
spec:
  bfd:
    minRX: 1000
    minTX: 1000
```

Increasing the interval reduces sensitivity to brief scheduling delays and packet loss, but increases failure detection time. With a 1000 ms interval and a `multiplier` of `5`, BFD takes approximately 5 seconds to detect a failure, excluding the time OVN needs to update the forwarding path.

## Scenario 2: No BFD sessions, but the health check succeeds

**Applicable versions:** ACP v4.3.0, v4.3.1, and v4.3.2

### Symptoms

After BFD is enabled for the VEG, `bfdd-control status` reports `There are 0 sessions:`, but the VEG Pod remains `Running` and `Ready`. The corresponding VEG next hop can become unavailable and interrupt application outbound traffic.

### Cause

The existing `bfdd` health check runs `bfdd-control status` directly. The command returns exit code `0` when it successfully communicates with the local daemon, even when no BFD sessions exist:

| BFD state | Example output | Exit code |
| --- | --- | --- |
| A session exists | `There are 1 sessions:` | `0` |
| No session exists | `There are 0 sessions:` | `0` |

Kubernetes exec probes evaluate only the command exit code and do not interpret its output. Therefore, the existing liveness probe does not treat an empty session table as a failure, and the kubelet does not restart the `bfdd` container.

### Resolution

Upgrade to ACP v4.3.3 or later. These versions treat an empty BFD session table as a health-check failure.

If the cluster must remain on ACP v4.3.0, v4.3.1, or v4.3.2, use the following temporary workaround.

:::warning
Updating the Deployment recreates VEG Pods. For a single-replica VEG, schedule a maintenance window because application outbound traffic can be interrupted during the rollout.

The VEG Deployment is generated by the Kube-OVN controller. This manual change can be overwritten when the VEG specification or replica count changes, Kube-OVN is upgraded, or the VEG or Deployment is recreated.
:::

Set the VEG namespace and name, and obtain the generated Deployment name:

```bash
NAMESPACE="<veg-namespace>"
VEG_NAME="<veg-name>"

DEPLOYMENT="$(kubectl -n "${NAMESPACE}" get deployment \
  -l "ovn.kubernetes.io/vpc-egress-gateway=${VEG_NAME}" \
  -o jsonpath='{.items[0].metadata.name}')"
```

Edit the generated Deployment:

```bash
kubectl -n "${NAMESPACE}" edit deployment "${DEPLOYMENT}"
```

In the `bfdd` container, modify `exec.command` in the existing `livenessProbe` and set `timeoutSeconds` to `10`:

```yaml
livenessProbe:
  exec:
    command:
      - bash
      - -ec
      - 'output="$(bfdd-control status)"; ! grep -q "^There are 0 sessions:" <<< "${output}"'
  timeoutSeconds: 10
```

This change detects an empty session table and attempts recovery by restarting the `bfdd` container. If the session table remains empty after the restart, the change can only expose the failure; it cannot repair the BFD session. Investigate the OVN BFD peer, network path, and BFD configuration.
