# Downgrading the Kafka operator on a cluster
## Overview

Operator Lifecycle Manager (OLM) is designed around a forward-only upgrade graph and does not support an in-place downgrade of an operator subscription channel to an older release. To roll the Kafka operator (a Strimzi-based distribution) back to an earlier version on a cluster, the operator must be uninstalled and reinstalled at the desired version. The Kafka data plane (`Kafka`, `KafkaTopic`, `KafkaUser`, `KafkaConnect` and related custom resources) survives the operator removal as long as the CRDs and namespace data are not deleted, and the older operator picks the workloads back up after reinstall.

This article describes the downgrade procedure and the configuration items that must be captured beforehand so the cluster does not regress.

## Resolution

### Capture configuration before removing the operator

Any change made through the current operator (channel pinning, custom RBAC, image overrides, watched namespaces, custom logging) is rendered into the operator's `Subscription` and `OperatorGroup`. Older operator versions may not preserve those settings, so capture them before uninstalling:

```bash
kubectl get subscription <name> -n <ns> -o yaml > kafka-operator-sub.bak.yaml
kubectl get operatorgroup -n <ns> -o yaml > kafka-operatorgroup.bak.yaml
kubectl get csv -n <ns> -o yaml > kafka-csv.bak.yaml
```

Also record the customised images and parameters of the existing data-plane custom resources so they can be reapplied if the older operator strips unsupported fields:

```bash
kubectl get kafka -A -o yaml > kafka-cr.bak.yaml
kubectl get kafkatopic,kafkauser,kafkaconnect -A -o yaml > kafka-others.bak.yaml
```

### Uninstall the current operator

Removing the `Subscription` plus the `ClusterServiceVersion` (CSV) is enough — it leaves the CRDs and the data-plane workloads intact:

```bash
kubectl delete subscription <name> -n <ns>
kubectl get csv -n <ns> -o name | xargs -r kubectl delete -n <ns>
```

Wait for the operator pods to terminate:

```bash
kubectl get pods -n <ns> -w
```

Do **not** delete the CRDs; doing so would garbage-collect every Kafka cluster, topic and user in scope.

### Reinstall at the target older version

Create a new `Subscription` pointing at the desired channel (or a specific `startingCSV`):

```yaml
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: kafka-operator
  namespace: <ns>
spec:
  channel: <older-channel>
  installPlanApproval: Manual
  name: <package-name>
  source: <catalog-source-name>
  sourceNamespace: <catalog-source-namespace>
  startingCSV: <older-csv-name>
```

`installPlanApproval: Manual` lets the cluster operator approve only the desired version's `InstallPlan` and prevents OLM from advancing back to the latest release on the next reconciliation:

```bash
kubectl get installplan -n <ns>
kubectl patch installplan/<name> -n <ns> --type merge -p '{"spec":{"approved":true}}'
```

After the older CSV reaches `Succeeded`, the operator pod returns and reconciles the surviving `Kafka` custom resources.

### Reapply settings from the backup

Some operator versions strip fields that the newer release introduced. Diff the post-install resources against the backup and reapply any missing tuning:

```bash
kubectl diff -f kafka-cr.bak.yaml || true
kubectl apply -f kafka-cr.bak.yaml
```

### Caveats

- A downgrade across a major Kafka broker version is not safe: brokers do not understand log-format versions written by a newer release. Plan downgrades only between operator versions whose embedded broker image is compatible with the current `Kafka.spec.kafka.version` setting, and keep `inter.broker.protocol.version` and `log.message.format.version` aligned with the older release before starting.
- Custom RBAC granted to the operator's service account is regenerated by the new install and may need manual replay if the older release shipped narrower roles.

## Diagnostic Steps

1. Confirm the operator is at the expected version after install:

   ```bash
   kubectl get csv -n <ns> -o jsonpath='{range .items[*]}{.spec.displayName}{"\t"}{.spec.version}{"\n"}{end}'
   ```

2. Confirm the operator has resumed reconciliation of the existing Kafka clusters:

   ```bash
   kubectl get kafka -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}{"\t"}{.status.conditions[?(@.type=="Ready")].status}{"\n"}{end}'
   ```

3. Watch the operator log for any field-stripping warnings on first reconcile:

   ```bash
   kubectl logs -n <ns> deployment/<operator-deployment> --tail=200
   ```
