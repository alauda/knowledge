## Issue

A cluster operator needs to attribute a `CronJob` back to the principal that created it — either a human user authenticated against the cluster's identity provider, or a service account used by a pipeline. The `CronJob` spec itself does not record the creator: `kubectl get cronjob <name> -o yaml` shows the `metadata.creationTimestamp`, labels, and annotations, but no `createdBy` field. The Kubernetes object only carries the identity of the controller that last updated it, which is typically the CronJob controller itself, not the original author.

This matters during incident response (who scheduled the workload that spikes the cluster at 03:00?) and during compliance review (prove that every production `CronJob` was created by an approved pipeline identity).

## Root Cause

The only authoritative record of who called the API server is the **audit log** generated by `kube-apiserver`. Every `CREATE` / `UPDATE` / `DELETE` request is logged with the authenticated user info (`user.username`, `user.groups`, `user.extra`), the target resource (namespace, kind, name), and the verb. On ACP this audit stream is captured per control-plane node under `/var/log/kube-apiserver/audit.log`, rotated according to the platform's retention policy.

Two practical constraints follow:

- **If audit logging is disabled or the retention window has passed**, the creator is gone. There is no other place in the cluster where that information is kept.
- **The audit policy must capture the resource at RequestResponse or Metadata level**. A `None` rule for `batch/v1` objects drops the event before it reaches the log, so even recent `CronJob` creations would be invisible.

When the audit log is available, the creator's `user.username` either takes the form `system:serviceaccount:<namespace>:<name>` (machine identity from a pipeline or operator) or a plain string that matches the identity-provider-projected name (human user).

## Resolution

Rather than SSHing into a control-plane node — which may not be allowed on ACP clusters that treat control planes as managed — prefer to collect the audit events through a forwarder that the platform already supports. When those paths are blocked, the in-pod debug fallback still works.

1. **Preferred — query the audit stream via `observability/log` or the Logging Service.** ACP's log pipeline ingests control-plane audit events (forwarded through Vector or the Cluster Log Forwarder-equivalent) into Loki. A canonical LogQL expression to find the creator of a specific `CronJob`:

   ```text
   {component="kube-apiserver", log_type="audit"}
     | json
     | verb = "create"
     | objectRef_resource = "cronjobs"
     | objectRef_name = "<cronjob-name>"
     | objectRef_namespace = "<namespace>"
     | line_format "{{.requestReceivedTimestamp}} {{.user_username}}"
   ```

   The output is one line per creation event, with the principal that issued the request.

2. **Fallback — run a debug pod on a control-plane node and grep the raw audit log.** This path is useful when log forwarding is not configured, or when only the raw JSON record (full request, response status) is needed:

   ```bash
   kubectl debug node/<control-plane-node> -it --image=registry/debug:latest -- chroot /host bash
   # inside the debug shell
   grep '"verb":"create"' /var/log/kube-apiserver/audit.log \
     | grep '"resource":"cronjobs"' \
     | grep '"name":"<cronjob-name>"' \
     | jq '{time:.requestReceivedTimestamp, user:.user.username, groups:.user.groups, sa:.user.extra}'
   ```

   `kubectl debug node/` attaches a privileged debug container to the node host namespace, so `chroot /host` exposes the node filesystem where the audit log lives.

3. **Interpret the `user.username` field.** The value always follows one of two shapes:

   - **Service account:** `system:serviceaccount:<namespace>:<name>`. The pipeline that owns this service account is the creator; cross-reference with GitOps repo or operator ownership to reach the human behind it.
   - **Human user:** whatever the identity provider stamps — typically an email address or corporate username. Check the `user.groups` field on the same audit record to confirm the principal's role at the time of creation.

4. **If the creator information is already gone**, there is no retrospective fix — the audit window has elapsed. Tighten the retention policy going forward:
   - Extend the audit log retention (`maxAge`, `maxBackups`, `maxSize` on the kube-apiserver configuration) so that last-week creations are still queryable.
   - Forward audit events into Loki through the Logging Service with a retention policy that matches your compliance window.
   - Ensure the audit policy does **not** drop `batch/v1` events with a `level: None` rule.

## Diagnostic Steps

Verify that audit events for the `CronJob` kind are actually being logged:

```bash
kubectl get --raw /metrics | grep 'apiserver_audit_event_total' | head
kubectl get --raw /metrics | grep 'apiserver_audit_level'
```

The `apiserver_audit_event_total` counter climbs on every request that matches an enabled audit level; if it never increments for `resource="cronjobs"`, the audit policy is dropping those events.

If using the log-forwarder path, sanity-check the stream label the forwarder attaches to the audit events:

```text
{component="kube-apiserver"} |= "cronjobs"
```

A small non-zero match rate over the last 15 minutes means the forwarder is healthy. An empty result with a green status page means the audit log is rotated or the `log_type` label differs from this example — inspect the forwarder's pipeline config.

If using the in-pod debug fallback, confirm the `kubectl debug node/` session actually landed on the right control-plane host and that the audit file is being rotated:

```bash
ls -lh /var/log/kube-apiserver/audit*.log
ls -lh /var/log/kube-apiserver/audit*.log.gz
```

Old rotations under `.gz` must be decompressed and grepped separately (`zgrep` rather than `grep`). If the file size is zero or the directory is missing, the control plane runs with audit logging disabled and no retroactive lookup is possible.
