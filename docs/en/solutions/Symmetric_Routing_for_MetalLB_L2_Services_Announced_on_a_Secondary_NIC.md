## Issue

A LoadBalancer Service is announced via MetalLB in L2 mode, but the VIP belongs to a subnet that lives on a *secondary* NIC of the announcing node, not on the cluster's primary `br-ex` bridge. From inside the secondary network, clients can reach the VIP. From any external network reached through the **default** gateway, the SYN packets arrive at the node but never receive a SYN-ACK; the connection times out.

Symptoms in a packet capture on the announcing node:

```text
In IP <client>.<port> > <vip>.443: Flags [S], seq xxx
In IP <client>.<port> > <vip>.443: Flags [S], seq xxx
In IP <client>.<port> > <vip>.443: Flags [S], seq xxx
# no matching outgoing SYN-ACK
```

A `curl` from a host on the secondary network to the VIP works:

```text
$ curl -vk https://<vip>/healthz
* Connected to <vip> (<vip>) port 443
< HTTP/1.1 200 OK
```

A `curl` from a host on any other external network reaches the node but never gets a reply. Adding static routes pinning the VIP's reverse traffic to the secondary NIC restores connectivity.

## Root Cause

A node with the OVN-based CNI has exactly one default gateway, owned by the primary bridge `br-ex`. When MetalLB advertises the VIP via L2 and the VIP's IP belongs to a subnet that overlaps the primary bridge's network, the kernel happily replies through `br-ex` and the connection completes.

When the VIP belongs to a subnet that is reachable only via a **non-primary** NIC, the asymmetry kicks in:

- Inbound SYN to the VIP arrives on the secondary NIC (because L2 ARP/NDP advertises the MAC of that NIC).
- Reply SYN-ACK is generated by the host stack and routed using the *system* routing table — which has its default route pointing at `br-ex`.
- The reply leaves through `br-ex`, sourced from the VIP. Most upstream routers (and reverse-path-filtering on the client) drop a packet whose source address belongs to a subnet that is not behind the interface it appeared on.

The platform CNI's `routingViaHost=true` mode keeps Service traffic on the host network namespace, so this is a host-routing problem, not a CNI problem. Until the platform exposes a first-class "egress symmetry per Service" knob, the fix is to install per-source policy routing on the announcing node so reply traffic for the VIP leaves through the same NIC the request arrived on.

## Resolution

ACP delivers MetalLB-style L2 advertisement through the `networking` capability area; node-network configuration is owned by `configure/clusters/nodes` (NMState) and Service-level egress shaping is owned by an `EgressService`-style CRD. The platform-preferred approach is to model the route policy declaratively rather than push static routes onto the node by hand.

### Option A — Static per-route on the secondary NIC (supported, simplest)

For one or two LoadBalancer Services, an explicit static route per VIP is the cleanest:

```yaml
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
  name: vip-static-route
spec:
  nodeSelector:
    role.example.com/lb-announcer: "true"
  desiredState:
    routes:
      config:
        - destination: <client-cidr>/<mask>
          next-hop-address: <gw-on-secondary>
          next-hop-interface: <secondary-nic>
          table-id: 254
```

This works because each `<client-cidr>` reaches the node through the secondary NIC, so its reply goes back the same way. The downside is one route per remote subnet; for hundreds of clients this becomes a burden.

### Option B — Source-based policy routing tied to an EgressService (declarative, scalable)

For many Services or many client subnets, install a dedicated routing table on the secondary NIC and bind the VIP to it through an `EgressService`. The platform provides this as an opt-in advanced configuration; treat it as not-yet-stable for production until validated against the platform's release notes.

1. **Provision the secondary NIC and a dedicated routing table.** A NodeNetworkConfigurationPolicy creates the bridge, the VLAN sub-interface, the IP, and a routing table with a default route — plus a route rule that picks the table whenever a packet is marked with the agreed firewall mark:

   ```yaml
   apiVersion: nmstate.io/v1
   kind: NodeNetworkConfigurationPolicy
   metadata:
     name: vip-secondary-nic
   spec:
     nodeSelector:
       role.example.com/lb-announcer: "true"
     desiredState:
       interfaces:
         - name: bond0.<vlan>
           type: vlan
           state: up
           vlan:
             base-iface: bond0
             id: <vlan>
         - name: br-vip
           type: ovs-bridge
           state: up
           bridge:
             options:
               stp: true
             allow-extra-patch-ports: true
             port:
               - name: bond0.<vlan>
         - name: vip-iface
           type: ovs-interface
           state: up
           ipv4:
             enabled: true
             address:
               - ip: <node-ip-on-secondary>
                 prefix-length: <mask>
           ipv6:
             enabled: true
             address:
               - ip: <node-v6>
                 prefix-length: <mask>
       routes:
         config:
           - destination: 0.0.0.0/0
             next-hop-address: <gw-on-secondary>
             next-hop-interface: vip-iface
             table-id: <table>          # e.g. matches VLAN id
       route-rules:
         config:
           - family: ipv4
             fwmark: "0x3f0"
             fwmask: "0xffffffff"
             priority: 4999
             route-table: 254           # leave 254 for default lookup; the EgressService writes to <table>
   ```

2. **Bind the LoadBalancer Service to the same table via an `EgressService`.** The platform's egress-service CRD watches for traffic sourced from the LoadBalancer IP and routes it through the named table:

   ```yaml
   apiVersion: k8s.ovn.org/v1
   kind: EgressService
   metadata:
     name: <same-name-as-the-LB-service>
     namespace: <ns>
   spec:
     network: "<table>"
     nodeSelector:
       matchLabels:
         role.example.com/lb-announcer: "true"
     sourceIPBy: LoadBalancerIP
   ```

   The `EgressService` name must equal the LoadBalancer Service name. The `nodeSelector` must overlap with the announcer node selected by the L2 advertisement, otherwise the policy applies on the wrong node and reply traffic still leaves via the default gateway.

3. **Verify the rules are installed on the announcing node.** New `ip rule` lines should appear that send traffic from the VIP into the dedicated table:

   ```bash
   NODE=$(kubectl get nodes -l role.example.com/lb-announcer=true -o name | head -1 | cut -d/ -f2)
   kubectl debug node/$NODE -it --image=registry.k8s.io/e2e-test-images/busybox:1.36 \
     -- chroot /host ip rule
   ```

   Expected entries include something like `5000: from <vip> lookup <table>`. If the rules are missing, the egress-service controller has not reconciled — check its logs.

### Picking between A and B

- For 1–10 client subnets reaching a small number of VIPs: Option A (static routes per remote network).
- For dynamic remotes or many VIPs on the same secondary NIC: Option B (per-source policy routing + `EgressService`).
- For environments where BGP is available, advertising the VIPs via BGP from the secondary NIC eliminates the asymmetry entirely (the same NIC that announces the route receives the reply). When the platform supports a BGP advertisement mode, prefer that over either option above.

## Diagnostic Steps

Identify the announcing node and confirm the L2 advertisement exists:

```bash
kubectl describe svc <name> -n <ns> | grep -E "LoadBalancer Ingress|nodeAssigned"
kubectl -n metallb-system get l2advertisement -o wide
```

Capture traffic on the announcing node to see whether reply packets are leaving:

```bash
NODE=<announcing-node>
kubectl debug node/$NODE -it --image=registry.k8s.io/e2e-test-images/busybox:1.36 -- \
  chroot /host tcpdump -nni any -y LINUX_SLL2 port <port>
```

From an *internal* client (same subnet as the VIP), confirm the connection works:

```bash
curl --local-port <high-port> -vk https://<vip>/healthz
```

From an *external* client (reaching the node through the default gateway), confirm the connection times out and that the capture shows inbound SYNs without matching outbound SYN-ACKs. That asymmetry is the fingerprint of the bug.

After applying Option A or B, repeat the capture: a healthy fix shows matched SYN/SYN-ACK pairs both leaving via the secondary NIC (`tcpdump -i <secondary-nic>`) and *not* via `br-ex`. If reply packets still come out of `br-ex`, the EgressService binding did not take — verify its `nodeSelector` and that the announcing node carries the matching label.
