# Node Pool Degraded by /etc/resolv.conf Content Mismatch
## Issue

One or more node pools in the cluster report `Degraded=True` after a node-configuration change that touches `/etc/resolv.conf`. The pool status carries a per-node message of the form `unexpected on-disk state validating against rendered-<pool>-<hash>`, and the on-node configuration daemon log expands that into a content diff:

```text
content mismatch for file /etc/resolv.conf (-want +got):
- "search example.com",
- "nameserver 10.0.0.1",
+ "# Generated by NetworkManager",
+ "search example.com",
+ "nameserver 10.0.0.2",
Marking Degraded due to: unexpected on-disk state
  validating against rendered-worker-xxxxxxx
```

The platform's declarative node-configuration surface (`configure/clusters/nodes`, backed by the **Immutable Infrastructure** extension) believes it owns the file, but the version on disk has been silently rewritten between reconciles. While the pool is degraded, no further configuration updates roll out.

## Root Cause

`/etc/resolv.conf` on the node OS is managed by NetworkManager, which regenerates it from active connection profiles whenever DHCP renews, an interface comes up, or `nm-online` fires. The first line of the rewritten file (`# Generated by NetworkManager`) is the giveaway in the daemon's diff.

The on-node configuration daemon validates the rendered config against on-disk state every reconcile loop. When NetworkManager overwrites a file the daemon thinks it owns, the byte-level comparison fails and the daemon refuses to proceed — propagating any later mistake on top of unknown drift would be far worse than stopping. As a result, declaring DNS servers through a node-configuration object on a node where NetworkManager is active is fundamentally racy: the platform writes the file, NetworkManager rewrites it on the next event, the next reconcile flags drift.

The supported way to set DNS for cluster nodes is at the source — DHCP options, NetworkManager connection profiles, or a sidecar that programs `/etc/NetworkManager/conf.d/` — rather than overwriting the rendered output.

## Resolution

1. **Locate the configuration object responsible for the conflict.** List all non-rendered configuration objects and find the one carrying a `/etc/resolv.conf` entry:

   ```bash
   kubectl get machineconfigs.machineconfiguration.k8s.io \
     | grep -v ^rendered-
   for mc in $(kubectl get machineconfigs.machineconfiguration.k8s.io \
                 -o jsonpath='{.items[*].metadata.name}' \
                 | tr ' ' '\n' | grep -v ^rendered-); do
     kubectl get machineconfigs.machineconfiguration.k8s.io $mc \
       -o yaml | grep -q '/etc/resolv.conf' && echo "owner: $mc"
   done
   ```

2. **Remove the resolv.conf entry from that object.** If the configuration object only managed `/etc/resolv.conf`, delete it outright:

   ```bash
   kubectl delete machineconfigs.machineconfiguration.k8s.io <name>
   ```

   If it carries other settings worth keeping, edit it and drop just the `storage.files` entry whose `path: /etc/resolv.conf`:

   ```bash
   kubectl edit machineconfigs.machineconfiguration.k8s.io <name>
   ```

   The rollout controller renders a fresh pool config and starts uncordoning nodes once each one passes validation.

3. **Re-establish DNS through a NetworkManager-friendly path.** Pick whichever fits the environment:

   - **DHCP option 6 / option 15** for clusters whose nodes pull addresses from a managed DHCP server. This is the cleanest option — every interface that comes up inherits the right resolver.
   - **A NetworkManager drop-in connection** delivered through the platform's node-configuration surface, written under `/etc/NetworkManager/conf.d/` rather than directly to `/etc/resolv.conf`. NetworkManager merges those settings when it next regenerates the file, so there is no fight between the two writers.
   - **`dnsmasq` configuration** if the node OS has it enabled, again expressed as a drop-in fragment rather than a wholesale resolv.conf overwrite.

4. **Wait for the pool to clear `Degraded`.** After the offending object is gone and a renewal pass runs:

   ```bash
   kubectl get machineconfigpools.machineconfiguration.k8s.io
   ```

   `Degraded` should flip to `False` and `UpdatedMachineCount` should climb to match `MachineCount` over the next few minutes. If a node stays degraded, its on-disk state still does not match the new rendered config — re-check the daemon log on that specific node.

## Diagnostic Steps

Identify the degraded pool and the affected nodes:

```bash
kubectl get machineconfigpools.machineconfiguration.k8s.io
kubectl get machineconfigpools.machineconfiguration.k8s.io <pool> -o yaml \
  | sed -n '/conditions:/,/^[^ ]/p'
```

Inspect the on-node daemon log for the precise file conflict:

```bash
kubectl -n kube-system get pods -l app=node-config-daemon -o wide
kubectl -n kube-system logs <daemon-pod-on-affected-node> -c node-config-daemon \
  | grep -E 'content mismatch|Marking Degraded'
```

Confirm the resolver file is really being rewritten by NetworkManager rather than something else (a hand-edited file, a CSI-mounted overlay):

```bash
NODE=<affected-node>
kubectl debug node/$NODE -it \
  --image=registry.k8s.io/e2e-test-images/busybox:1.36 \
  -- chroot /host /bin/sh -c '
    head -n 2 /etc/resolv.conf
    nmcli -t -f IP4.DNS,IP6.DNS device show
  '
```

A first line containing `Generated by NetworkManager` and `nmcli` reporting the live DNS servers nail the source of the rewrite. After the resolver settings move to a NetworkManager-aware path, the daemon's next reconcile log line should be `Update completed for config <name>` and the diff line should be gone.
