## Issue

Pods on a node lose network connectivity in short bursts; the host's physical interfaces flap up and down on their own; and the kernel ring buffer carries entries from the Intel ICE driver of the form:

```text
ice 0000:<bdf>: Failed stop Rx rings, VSI error-110
```

The interfaces recover automatically afterwards — the driver tears the VSI down and rebuilds it, the link comes back, and pod traffic resumes — but the gap (anywhere from a couple of seconds to a minute, depending on the depth of the rebuild) is enough to drop in-flight TCP connections, age out service-mesh probes, and trigger flaky readiness on workloads that share that NIC. The pattern repeats: a few minutes of healthy traffic, then another `error-110`, then another rebuild.

## Root Cause

`-110` is `ETIMEDOUT`. The driver asked the firmware to stop the receive rings on a Virtual Station Interface (VSI), the firmware did not acknowledge within the driver's deadline, and the driver gave up. Because a VSI cannot be left half-shut, the driver's recovery path is to reset the entire NIC and rebuild the VSI tree from scratch — that is the link flap and the connectivity gap operators see.

The timeout itself is not generated by the kernel or by the workload. It comes from the firmware-to-driver mailbox missing a response, which in practice is one of:

- A firmware bug that has been fixed in a later release for the same NIC SKU.
- A specific traffic pattern (high small-packet rates on certain queue counts, mixed VF/PF teardown ordering) that the firmware does not handle correctly.
- A hardware fault on the NIC or its PCIe slot — rarer, but the symptom is identical.

Distinguishing them is firmware-version work, not Kubernetes work. The cluster layer's role is to localize the problem to a specific NIC on a specific node and capture the artifacts the hardware vendor needs.

## Resolution

The fix is on the host, not on Kubernetes objects. Order:

### Capture the artifact

Before doing anything that will reboot the node, dump the kernel ring buffer and the driver/firmware versions. These are what the NIC vendor will ask for.

```bash
kubectl debug node/<node> -it --profile=sysadmin --image=<utility-image> \
  -- chroot /host bash -c '
    dmesg -T | grep -E "ice|i40e" | tail -200 > /tmp/ice-dmesg.txt
    ethtool -i <iface>                         > /tmp/ice-driverinfo.txt
    ethtool -S <iface>     | grep -i error     > /tmp/ice-counters.txt
    lspci -vvv -s <bdf>    | head -40          > /tmp/ice-pci.txt
  '
```

### Update the NIC firmware

The supported remediation for `Failed stop Rx rings, VSI error-110` is to bring the NIC's firmware to the latest release that the hardware vendor publishes for the SKU, and to confirm the in-tree `ice` driver version on the running kernel is compatible with that firmware. Use the vendor's NVM update package (typically `nvmupdate64e` from Intel, or the OEM's wrapper of it). The update runs from a host shell and requires the interface to be down for the duration:

```bash
# On the affected node, in the host namespace:
ip link set <iface> down
./nvmupdate64e -u -l -c nvmupdate.cfg          # vendor tool
ip link set <iface> up
ethtool -i <iface>                              # confirm new fw-version
```

The flash is an in-place hardware update; treat it as a node maintenance operation. Cordon and drain the node first so workloads are off the affected NIC before the link goes down:

```bash
kubectl cordon <node>
kubectl drain <node> --ignore-daemonsets --delete-emptydir-data --disable-eviction
```

After the firmware is updated and the link is up, re-check `dmesg` for ICE entries over a working window — at least a few hours of representative traffic. If `error-110` does not return, uncordon:

```bash
kubectl uncordon <node>
```

### When firmware does not fix it

A persistent `error-110` after the latest firmware narrows the problem to two remaining causes — a firmware bug not yet released, or a hardware fault. Both require the hardware vendor's support channel. Prepare:

- The dmesg / ethtool / lspci dump captured above.
- The NIC's part number, serial, and slot.
- A reproduction window (start/end timestamps) that matches an `error-110` line in `dmesg`.

Until the vendor responds, mitigate at the cluster level. Two options that buy time:

- Cordon the node so the scheduler avoids it; for workloads that already tolerate node loss, this is the cheapest mitigation.
- If the cluster has bonded interfaces or multiple NICs per host, fail traffic to a different physical port by adjusting the bond's primary or by moving the host's default route — the ICE NIC will still flap, but pod traffic no longer rides over it.

## Diagnostic Steps

1. Localize the failure to the right node and the right interface. The `dmesg` line identifies the PCI BDF; map BDF → interface name with `ethtool` or `ip link`:

   ```bash
   kubectl debug node/<node> -it --profile=sysadmin --image=<utility-image> \
     -- chroot /host bash -c '
       dmesg -T | grep -E "ice .* Failed stop Rx rings.*error-110" | tail -10
       for nic in /sys/class/net/*; do
         iface=$(basename "$nic")
         drv=$(readlink -f "$nic/device/driver" 2>/dev/null | xargs -r basename)
         bdf=$(readlink -f "$nic/device" 2>/dev/null | xargs -r basename)
         [ "$drv" = ice ] && echo "$iface  $bdf"
       done
     '
   ```

2. Confirm the firmware and driver versions before any change — the vendor's bug tracker indexes them as a pair:

   ```bash
   ethtool -i <iface>
   # driver: ice
   # version: <kernel-driver-version>
   # firmware-version: <nvm><package> <eetrack>
   ```

3. Watch error counters over a workload window. The ICE driver exposes per-queue counters; spikes on `rx_*_errors` or `tx_*_errors` correlate one-to-one with the dmesg line:

   ```bash
   ethtool -S <iface> | grep -E '(error|drop|miss|timeout|hang)' | sort
   ```

4. Quantify the pod-traffic impact before the fix is deployed — that is the metric the rebuild has to improve. From a control plane host:

   ```bash
   kubectl get events -A --field-selector reason=Unhealthy \
     -o jsonpath='{range .items[*]}{.lastTimestamp}  {.involvedObject.namespace}/{.involvedObject.name}  {.message}{"\n"}{end}' \
     | grep <node> | sort | tail -30
   ```

   Probe failures clustered around `dmesg` `error-110` timestamps confirm the link flap is what the workload is feeling.

5. After the firmware update lands, leave a `dmesg --follow | grep -E "ice|error-110"` running on the node for a representative period (matching the longest previous quiet window before recurrence) before declaring the issue closed. The failure is intermittent; absence over a short window does not prove a fix.
