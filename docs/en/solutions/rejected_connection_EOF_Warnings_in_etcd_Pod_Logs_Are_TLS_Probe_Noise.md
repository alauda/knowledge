# etcd "rejected connection" (server name doesn't match certificate) on ACP

## Issue

On Alauda Container Platform clusters running etcd as a kubeadm-style static pod (`etcd-<control-plane-IP>` in `kube-system`, image `registry.alauda.cn:60080/tkestack/etcd:v3.5.28-260325`, kube `v1.34.5`), the etcd pod log contains warnings whose `msg` is `rejected connection on client endpoint`. Each line is emitted by etcd's own embed package and shows up as a structured JSON record with `level=warn`, `caller=embed/config_logging.go:177`, a `remote-addr` for the connecting peer, a `server-name` field, and an `error` describing why the inbound connection was dropped before any application bytes were exchanged.

A representative log entry has this shape:

```text
{"level":"warn","caller":"embed/config_logging.go:177","msg":"rejected connection on client endpoint","remote-addr":"127.0.0.1:33772","server-name":"","error":"..."}
```

The `remote-addr` field records the source IP and source port of the connecting peer in `ip:port` form, which makes it the primary handle for identifying who initiated the dropped connection.

## Root Cause

The warning is generated by etcd itself, inside its TLS listener wiring in the `embed` package, when an incoming TCP+TLS connection is closed before the server can read any application data. The empty `server-name=""` field on these lines indicates the connecting TLS client did not send a Server Name Indication (SNI) extension in its ClientHello — etcd records the SNI it received (or the empty string if none was offered) and includes it in the reject log so the operator can see what the connecting client presented at handshake time.

On this ACP control plane, the principal TLS client of etcd is the kube-apiserver running as a static pod (`kube-apiserver-<control-plane-IP>` in `kube-system`); each control-plane node runs one kube-apiserver static pod, and that apiserver is etcd's client for CRUD and watch traffic. On a single-control-plane cluster the `remote-addr` values come back as `127.0.0.1:<ephemeral-port>`; on a multi-master cluster the warnings normally carry the peer control-plane node IPs, reflecting connections from the kube-apiserver instances on the other masters.

Note that `remote-addr=127.0.0.1` only establishes that the connection originated from a process *on the same host* — it does not by itself prove the source is kube-apiserver. Any local client (a health probe, a backup job, an `etcdctl` invocation, a monitoring sidecar) opening a TLS connection to `127.0.0.1:2379` and then closing it before completing the handshake produces an identically shaped line. The kube-apiserver is the expected and dominant local client on a control-plane node, so it is the first candidate, but confirm attribution by correlating the source port with the owning process (see Diagnostic Steps) rather than inferring it from the loopback address alone.

## Resolution

When this warning is present in isolation — that is, etcd has no active alarms and its endpoint health remains healthy — the warning by itself does not require operator action and does not degrade the cluster. On the test environment, even with 121 `rejected connection` warnings present in the etcd log, `etcdctl alarm list` returned no rows and the endpoint reported healthy with a commit-proposal latency around 11.6 ms, well inside the normal range.

Run a quick health sanity check before treating these warnings as actionable:

```bash
kubectl -n kube-system exec etcd-<control-plane-IP> -- etcdctl \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  --endpoints=https://127.0.0.1:2379 \
  endpoint health
```

```bash
kubectl -n kube-system exec etcd-<control-plane-IP> -- etcdctl \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  --endpoints=https://127.0.0.1:2379 \
  alarm list
```

If endpoint health is `is healthy` and `alarm list` returns no rows, the `rejected connection` warning is informational and the cluster can continue to operate without intervention.

## Diagnostic Steps

Retrieve the warnings directly from the etcd container's logs in `kube-system` and filter for the literal `rejected connection` string. The etcd static pod is named `etcd-<control-plane-IP>` (one per control-plane node) — substitute the node IP for the cluster being inspected:

```bash
kubectl logs -n kube-system etcd-<control-plane-IP> | grep 'rejected connection'
```

For example, on a single-master ACP test cluster with control-plane IP `192.168.135.152`:

```bash
kubectl logs -n kube-system etcd-192.168.135.152 | grep -i 'rejected connection'
```

Each matching line is a structured JSON record; the load-bearing fields for triage are `remote-addr`, `server-name`, and `error`.

Inspect `remote-addr` to identify the connecting peer. The value follows `ip:port`. On a single-control-plane cluster the values are `127.0.0.1:<ephemeral-port>`, which narrows the source to a process on the same host — most likely the local kube-apiserver, but not provably so from the address alone. On a multi-master cluster a non-loopback IP maps to another control-plane node, pointing at the kube-apiserver static pod (`kube-apiserver-<control-plane-IP>` in `kube-system`) on that peer.

To confirm which local process owns a loopback connection rather than assuming it is kube-apiserver, correlate the source port from `remote-addr` with the owning process on the node. Open a host shell on the control-plane node (e.g. via `kubectl debug node/<name>` or the platform's node-access path) and match the connection's source port to a PID and command:

```bash
ss -tnp | grep ':2379'
```

The row whose local/peer port matches the `remote-addr` port shows the owning process in the `users:(("<comm>",pid=<pid>,...))` field; expect `kube-apiserver` for routine control-plane traffic, and treat any other command as the actual source for that line. Because the source ports are short-lived, run this while the connections are active rather than after the fact.

When `server-name` is the empty string on a given record, the connecting client opened the TLS handshake without offering an SNI value in its ClientHello — this is normal for etcd's internal client traffic and is reported verbatim by etcd's embed listener.

Cross-check etcd health to rule out a real outage masquerading as a noisy log:

```bash
kubectl -n kube-system exec etcd-<control-plane-IP> -- etcdctl \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  --endpoints=https://127.0.0.1:2379 \
  endpoint health
```

A healthy endpoint with no active alarms, paired with these `rejected connection` warnings, is consistent with the benign condition described under Resolution.
