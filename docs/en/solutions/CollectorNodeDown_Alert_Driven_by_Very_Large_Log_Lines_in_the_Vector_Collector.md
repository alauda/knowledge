## Issue

The logging stack raises a `CollectorNodeDown` alert against one or more Vector collector pods. On closer inspection the collector is not *actually* down — it is alive and scheduled — but it is saturating its CPU budget to the point that it stops reporting scrape health in time, so the monitoring rule flips the alert.

The same nodes also show a more subtle symptom at the application end: a subset of **very large** log lines (on the order of several megabytes per line) never arrive at the configured log destinations. Lines from the same application that are of normal size get through, so it is not a destination-connectivity problem.

Temporarily raising the CPU limit on the collector DaemonSet clears the alert for a while, then it returns.

## Root Cause

Two behaviours compound:

1. Vector's application-log `source` (the container-log reader) ingests each log line into its internal buffer before the pipeline decides what to do with it. When a line is several megabytes, the per-line cost in allocation, UTF-8 validation, JSON parsing (for the pipeline's intermediate representation), and downstream encoding is large enough that the component stalls the pipeline while it processes it.

2. The downstream pipeline has a finite buffer. When that buffer fills because of the stalled big-line processing, Vector applies back-pressure up-chain. Under sustained back-pressure the collector's CPU usage spikes as it tight-loops on buffer state, and on a cluster with many large lines in flight the collector never drains fast enough to keep its health endpoint responsive — which is what `CollectorNodeDown` actually fires on.

Secondary consequence: some very large lines are silently dropped mid-pipeline under extreme pressure, so destinations are missing them. The user-visible picture is "alert + missing logs".

Upstream engineering is tracking the underlying bug in Vector; until the behaviour is fixed, the practical mitigation is to prevent unreasonably large lines from entering the pipeline in the first place.

## Resolution

Apply a `maxMessageSize` to the `input` in the `ClusterLogForwarder` CR so that the collector enforces an upper bound on individual log-line size. Lines that exceed the bound are dropped (with a metric incremented) at ingestion, before they can stall the downstream buffer.

This is a workaround, not a root-cause fix — it trades a small amount of observability (the offending multi-megabyte lines are discarded) for stability of the collector. For any application that genuinely needs its full log record to reach the store, the right long-term answer is to teach the application not to emit megabyte-sized single lines (rotate large dumps into a file and stream them, or break the record into bounded chunks).

1. Set the environment variables you will need:

   ```bash
   export cr=<clusterlogforwarder-name>
   export ns=<logging-ns>
   ```

2. Open the `ClusterLogForwarder` for editing. The short name `obsclf` works on clusters where the logging operator registers it; otherwise address the CR by its full plural + group:

   ```bash
   kubectl -n "$ns" edit obsclf "$cr"
   ```

3. Declare (or modify) an `input` of type `application` with a `tuning.maxMessageSize`. The value is in bytes — three megabytes is a reasonable starting budget; lower it further only if the problem recurs:

   ```yaml
   spec:
     inputs:
     - name: app
       type: application
       application:
         tuning:
           maxMessageSize: 3000000
   ```

4. Reference that named input from the `pipeline` that should enforce the limit. A pipeline that uses the built-in `application` input directly does *not* get the tuning; you have to go through the named input:

   ```yaml
   spec:
     pipelines:
     - name: default-logstore
       inputRefs:
       - app
       outputRefs:
       - default-lokistack
   ```

5. Save and let the logging operator roll the Vector DaemonSet. Watch the alert clear and CPU drop back to baseline.

## Diagnostic Steps

1. **Confirm the CPU symptom.** The simplest view is `kubectl top`, scoped to the collector's namespace; the PromQL below is the equivalent that the alert rule uses, which is useful for confirming you are reading the same signal as the alert:

   ```bash
   kubectl top pods -n <logging-ns> | grep collector
   ```

   ```promql
   pod:container_cpu_usage:sum{pod=~'.*',namespace='<logging-ns>'}
   ```

2. **Confirm the missing-large-lines symptom.** Pick a known multi-megabyte line generated by the application (or generate one deliberately with a test pod) and search the destination for it. If the normal-size neighbours show up and the large line does not, that matches this failure:

   ```bash
   # in the destination log store's query surface, search by a unique token
   # that appears only in the large line
   ```

3. **Use `vector tap` to watch the pipeline** on the collector co-located with the application that emits the large line. The line should appear on the `input_application_container` component and then flow to the next stage; if it appears at `input_application_container` and never progresses, the big-line stall is confirmed:

   ```bash
   kubectl -n <logging-ns> exec -it <collector-pod> -- sh
   # inside the collector:
   vector tap --outputs-of input_application_container
   ```

4. **Validate the mitigation after applying `maxMessageSize`.** The large line should now be dropped cleanly at the `input` boundary (visible as an incremented discard metric), the downstream pipeline should no longer stall, and CPU should return to baseline:

   ```bash
   kubectl top pods -n <logging-ns> | grep collector
   kubectl -n <logging-ns> get clusterlogforwarder "$cr" -o yaml | yq '.status'
   ```

5. **Track the upstream fix.** Once the collector release that properly handles oversized lines is deployed in the cluster, revisit the `maxMessageSize` setting — you may be able to remove it, or at least raise it so fewer useful records are discarded.

If the alert does not clear after the mitigation, check for a second, unrelated CPU driver on the same collector (a too-permissive `multiline` regex that causes excessive buffering, a misbehaving `remap` transform, or a destination that is throttling writes and sending back-pressure) — those look similar from the top-level alert but have different fixes.
