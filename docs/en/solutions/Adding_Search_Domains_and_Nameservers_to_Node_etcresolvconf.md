## Issue

A workload migrated into the cluster fails to reach external hosts because the application sends short, unqualified names that the cluster's default DNS chain does not resolve. Operators reach for `/etc/resolv.conf` on the worker nodes — to add a search domain so that `db` resolves to `db.example.com`, or to point at a different nameserver — and find that any hand-edit is reverted within minutes by `NetworkManager`. Attempts to push the change through the platform's generic file-overlay surface trigger a node-pool reconcile failure that looks similar to:

```text
content mismatch for file /etc/resolv.conf (-want +got):
  - "search example.com"
  + "# Generated by KNI resolv prepender NM dispatcher script"
  + "search test.example.com"
  + "nameserver 5.6.7.8"
  ...
```

The pool drops to `Degraded`, no further reconciles land on it, and the cluster cannot upgrade until the conflict is resolved.

## Root Cause

`/etc/resolv.conf` on a cluster node is **not** a normal config file: it is owned by `NetworkManager`, which generates it from a combination of DHCP-supplied values, NM connection profiles, and dispatcher scripts. Anything that writes to the file directly (a generic file-overlay, `MachineConfig`-style raw-file injection, a hand-edit) competes with `NetworkManager` for ownership of the same path; the next NM event re-renders the file and the node-pool reconciler then sees an unexpected on-disk state and refuses to proceed.

The supported way to change DNS on a node is to express the change as **node network state** — search domains, nameservers, options — rather than as raw file content. The platform-managed mechanism for that is the **NMState** declarative API (open-source upstream, integrated into the platform's node-configuration surface). NMState renders the desired DNS into NM connection profiles in the right order, and `NetworkManager` then writes a coherent `/etc/resolv.conf` that the file checker accepts.

## Resolution

Pick the change site by what is actually wrong. Do **not** start with `/etc/resolv.conf` directly under any circumstances.

### Preferred for environments using DHCP: fix it at the DHCP server

If the search list is wrong because the DHCP scope omits domains, set DHCP **Option 119** (search list) on the scope that serves the node network and let the nodes re-renew. This is one server-side change, applies to every node automatically, and avoids touching the cluster at all. Reach for an in-cluster override only when DHCP cannot be changed (third-party, frozen production scope, etc.).

### Preferred for declarative node DNS: NMState NodeNetworkConfigurationPolicy

ACP's `configure/clusters/nodes` surface integrates NMState. Add a `NodeNetworkConfigurationPolicy` (NNCP) that declares the desired DNS state, scoped to the relevant nodes via `nodeSelector`. NMState merges the change into the appropriate `NetworkManager` connection profiles; `NetworkManager` regenerates `/etc/resolv.conf`; the node-pool reconciler is happy because nothing fought it.

Example: add a search domain and a secondary nameserver to all worker nodes, while preserving the values supplied by DHCP:

```yaml
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
  name: workers-extra-dns
spec:
  nodeSelector:
    node-role.kubernetes.io/worker: ""
  desiredState:
    dns-resolver:
      config:
        search:
          - corp.example.com
          - svc.example.com
        server:
          - 10.40.0.10
          - 10.40.0.11
```

Apply with `kubectl apply -f workers-extra-dns.yaml`. The NMState handler picks the policy up, lays it down on each matching node in serial (no concurrent rollout per node), and reports success in `NodeNetworkConfigurationEnactment` (one per node + policy).

A few rules to keep this safe:

- **Scope by `nodeSelector`.** A policy with no selector targets every node in the cluster. For DNS, that is rarely intended.
- **Do not mix NMState DNS with raw file overlays for `/etc/resolv.conf`.** Pick one mechanism and stick with it. The two will fight every reconcile.
- **Validate the rendered file before declaring success.** NMState succeeds when it has written the NM profile; whether `/etc/resolv.conf` then contains what you expected depends on NM merging rules (DHCP-supplied entries can interleave). Always verify on a node — see Diagnostic Steps.

### Per-pod alternative when only one workload is affected

If only one application has the FQDN problem, change the pod's DNS rather than the node's. Set `dnsPolicy: None` and supply a custom `dnsConfig` on the PodSpec:

```yaml
spec:
  dnsPolicy: None
  dnsConfig:
    nameservers:
      - 10.40.0.10
    searches:
      - corp.example.com
      - svc.example.com
    options:
      - name: ndots
        value: "2"
```

This avoids any node-level change and any reconcile risk. Prefer it whenever the requirement is scoped to a single workload.

## Diagnostic Steps

After applying an NNCP, watch the per-node enactment status:

```bash
kubectl get nodenetworkconfigurationenactment | grep workers-extra-dns
```

Each row shows the policy plus the node it landed on. The expected status is `Available`; `Failing` carries an error message that names the host and the conflicting setting. If a node sits in `Progressing` indefinitely, the rollout is stalled (often by a busy `NetworkManager` waiting on a DHCP lease) — re-check after five minutes before treating it as failed.

Confirm the node-pool itself is not degraded:

```bash
kubectl get nodes -o wide
```

If a node shows `SchedulingDisabled` for more than the rollout window, look at the platform-side rollout state for that pool; a degraded pool blocks every subsequent change, including unrelated ones.

Verify the rendered file on a representative node:

```bash
NODE=<worker-node>
kubectl debug node/$NODE -it \
  --image=registry.k8s.io/e2e-test-images/busybox:1.36 \
  -- chroot /host sh -c '
     cat /etc/resolv.conf
     echo "---"
     nmcli -t -f IP4.DNS,IP4.DOMAIN device show $(nmcli -g GENERAL.DEVICE dev | head -n1)
   '
```

The rendered `/etc/resolv.conf` should contain the search domains and nameservers from the NNCP. If it does not, but the enactment shows `Available`, the change has been written into a connection profile that is currently inactive — recreate the active connection (`nmcli con down <name>; nmcli con up <name>`) or wait for the next NM event to swap profiles.

To resolve a name from inside the cluster network and confirm the new DNS chain reaches the right server:

```bash
kubectl run dns-debug --rm -it --restart=Never \
  --image=registry.k8s.io/e2e-test-images/jessie-dnsutils:1.5 \
  -- nslookup db corp.example.com
```

A successful answer with the expected upstream IP confirms the search-domain/nameserver chain end to end. A `NXDOMAIN` here while the same query works on a node means the cluster DNS service has not picked up the change yet; bounce the cluster DNS pods and re-test.
