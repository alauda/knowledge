## Issue

A cluster has the NGINX Ingress Operator installed at an older release (for example, `v2.3.x`). A newer release (`v3.x`) is published in the same subscribed channel. Operator Lifecycle Manager (OLM) does not promote the cluster to the new release: no `InstallPlan` is created for `v3.x`, the `Subscription` keeps reporting the existing `currentCSV`, and the operator pod stays on the old version.

## Root Cause

OLM honours an explicit upgrade graph maintained by the operator vendor inside the operator catalog. For each `ClusterServiceVersion`, the catalog declares which previous CSVs can be replaced and through which channel. If the vendor has not published a `replaces`/`skips` entry connecting the older CSV to the new one — common when a release crosses a major version boundary that the vendor wants to gate on a manual migration — OLM has no way to construct a valid upgrade path. With no path, no `InstallPlan` is generated, and the existing CSV stays in place.

This is the case here: the vendor (NGINX) has not declared `v2.3.x` as replaceable by `v3.x` in the catalog, so OLM treats the two as parallel installations rather than an upgrade chain.

## Resolution

Move the cluster to the new major version by uninstalling the old operator and installing the new one. The data plane (`NginxIngressController` custom resources, `Service`, `Endpoints`, traffic-handling pods) survives the operator removal as long as the CRDs are not deleted, but a careful capture/restore of the workload custom resources is recommended in case the new operator's CRD shape changed.

### Procedure

1. Capture the existing custom resources so the configuration can be reapplied:

   ```bash
   kubectl get nginxingresscontroller -A -o yaml > nginx-ic.bak.yaml
   kubectl get subscription -n <operator-ns> -o yaml > nginx-sub.bak.yaml
   ```

2. Uninstall the operator. Removing the `Subscription` and the `ClusterServiceVersion` is enough; do **not** remove the CRDs — that would garbage-collect every controller managed by them:

   ```bash
   kubectl delete subscription nginx-ingress-operator -n <operator-ns>
   kubectl get csv -n <operator-ns> -o name | xargs -r kubectl delete -n <operator-ns>
   ```

3. Wait for the operator pods to terminate:

   ```bash
   kubectl get pods -n <operator-ns> -w
   ```

4. Install the new release. Either through the cluster's marketplace UI for the operator, or by creating a new `Subscription` that targets the desired CSV:

   ```yaml
   apiVersion: operators.coreos.com/v1alpha1
   kind: Subscription
   metadata:
     name: nginx-ingress-operator
     namespace: <operator-ns>
   spec:
     channel: alpha
     installPlanApproval: Manual
     name: nginx-ingress-operator
     source: <catalog-source>
     sourceNamespace: <catalog-source-ns>
     startingCSV: nginx-ingress-operator.v3.4.1
   ```

   With `installPlanApproval: Manual`, approve only the desired version's `InstallPlan` so OLM cannot bump the operator further on the next reconciliation:

   ```bash
   kubectl get installplan -n <operator-ns>
   kubectl patch installplan/<name> -n <operator-ns> --type merge -p '{"spec":{"approved":true}}'
   ```

5. Once the new CSV is `Succeeded`, re-apply the captured custom resources, adjusting any field renames or schema migrations called out in the new release's notes:

   ```bash
   kubectl diff -f nginx-ic.bak.yaml || true
   kubectl apply -f nginx-ic.bak.yaml
   ```

6. Validate that the new operator has reconciled the controllers and that traffic continues to flow as expected:

   ```bash
   kubectl get nginxingresscontroller -A
   kubectl get pods -n <controller-ns>
   ```

### Caveats

- A new major version may rename or drop fields in the `NginxIngressController` CRD. Keep the new operator scaled to zero until the resources have been adjusted to match the new schema, then restart the operator pod so it picks up the migrated objects cleanly.
- Custom RBAC granted to the operator's `ServiceAccount` is regenerated by the new install. Any cluster-administrator-issued bindings must be reapplied.
- If the cluster has multiple ingress controllers, time the migration during a low-traffic window — there is no in-place upgrade path that preserves the running pods across operator versions.

## Diagnostic Steps

1. Confirm no `InstallPlan` for the target version exists:

   ```bash
   kubectl get installplan -n <operator-ns>
   ```

2. Read the current subscription state to see why OLM has not advanced:

   ```bash
   kubectl get subscription nginx-ingress-operator -n <operator-ns> -o yaml | yq '.status'
   ```

   The `state` typically remains `AtLatestKnown` for the **older** CSV when no upgrade edge exists.

3. Inspect the catalog source's package manifest to confirm the available channels and their head CSVs:

   ```bash
   kubectl get packagemanifest nginx-ingress-operator -o yaml | yq '.status.channels[] | {name, currentCSV}'
   ```

4. After the reinstall, watch the new operator pod log for any reconciliation warnings about migrated custom resources:

   ```bash
   kubectl logs -n <operator-ns> deployment/nginx-ingress-operator --tail=200
   ```
