## Issue

A workload running inside a pod sometimes needs a fixed name-to-IP mapping that is not (or cannot be) served by cluster DNS — a third-party endpoint reachable only by an IP that DNS does not resolve, a temporary override during integration testing, or a legacy hostname expected by an application. Editing `/etc/hosts` inside the container by hand is not durable: the file is regenerated by the kubelet every time the pod restarts, and any in-place change vanishes.

## Root Cause

The kubelet owns `/etc/hosts` for every pod. On each restart it writes a fresh copy that contains:

- the pod's own IP (and short / FQDN names),
- standard `localhost` entries,
- entries for the pod's IPv6 stack, if applicable.

Anything written by the workload itself — by an init script, by `sed`, or by a sidecar — is overwritten. The only supported way to add entries is to declare them on the PodSpec; the kubelet then merges them into the generated file.

The PodSpec field is `spec.hostAliases`. This is part of upstream Kubernetes and is portable across container runtimes; it has been available since well before any current ACP release.

## Resolution

Declare the additional name/IP pairs under `spec.hostAliases` in the pod template of whatever controller manages the workload (Deployment, StatefulSet, DaemonSet, Job, or a bare Pod). The same shape applies in every case; only the path to the pod template differs.

Example, embedded inside a Deployment:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: example
  namespace: app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: example
  template:
    metadata:
      labels:
        app: example
    spec:
      hostAliases:
        - ip: "127.0.0.1"
          hostnames:
            - "home"
            - "local-shim"
        - ip: "10.40.5.12"
          hostnames:
            - "legacy-db"
            - "legacy-db.internal"
      containers:
        - name: app
          image: example.io/app:1.0
```

A few practical guard-rails:

- **Watch the indentation.** `hostAliases` belongs at `spec.template.spec.hostAliases` for any controller, not under `spec` of the controller itself. Misplacing it produces a silent no-op (the field is ignored by the API server).
- **Do not edit the pod template of operator-managed workloads in place.** A reconcile from the operator will revert the change. If an operator-managed CR offers a `podTemplate` extension or a similar pass-through, set the aliases through that surface instead. Otherwise, fork the manifest into a workload that the operator does not own.
- **Use `hostAliases` for static overrides only.** It is not a replacement for cluster DNS or for a sidecar resolver. If the set of names changes at runtime, route those names through CoreDNS (a custom zone, a forwarder, or a `hostname.namespace.svc.cluster.local` style entry).
- **For tweaking `/etc/resolv.conf`** (additional search domains, custom nameservers), use `spec.dnsConfig` and `spec.dnsPolicy` instead — that is a different field with its own semantics and is documented separately.

After updating the controller, the change applies on the next pod restart; existing pods are not patched in place.

## Diagnostic Steps

Inspect the rendered `/etc/hosts` inside a running pod to confirm the kubelet picked up the aliases:

```bash
kubectl -n <namespace> exec <pod> -- cat /etc/hosts
```

Expected output includes the kubelet-generated header (pod IP / pod name / loopback) followed by the entries declared in `spec.hostAliases`, each on its own line:

```text
# Kubernetes-managed hosts file.
127.0.0.1   localhost
::1         localhost ip6-localhost ip6-loopback
...
127.0.0.1   home  local-shim
10.40.5.12  legacy-db  legacy-db.internal
```

If the alias lines are missing, the most common causes are: the manifest was applied but the existing pod has not restarted, the field was placed at the wrong nesting level, or an admission controller stripped the field. Confirm the live spec with:

```bash
kubectl -n <namespace> get pod <pod> -o jsonpath='{.spec.hostAliases}{"\n"}'
```

An empty result on a pod that should have aliases means the PodSpec on disk no longer matches the controller — re-apply and let the controller create a new pod.
