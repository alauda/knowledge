# ACP 巡检 / 告警分级处置 Checklist

> 这个 checklist 的目标不是替代排障，而是帮支持现场先把问题分级：
>
> - **要立即处理**
> - **要尽快跟进**
> - **可继续观察**
> - **更像规则/模板/展示问题**

## 1. 先判断：这是哪一类信号？

- [ ] 巡检报告项
- [ ] 实时告警
- [ ] 历史告警
- [ ] 组件异常重启/状态异常
- [ ] 日志查询 / 展示异常
- [ ] apiserver / 认证类报错

## 2. 先做 6 个现场确认

- [ ] 是否已有真实业务影响（访问失败、延迟升高、错误率上升）
- [ ] 影响范围是单 Pod / 单节点 / 单集群 / 多集群
- [ ] 是持续异常还是瞬时波动
- [ ] 是否可复现
- [ ] 是否刚做过升级、变更、扩缩容、参数调整
- [ ] 当前异常是“采集/展示/通知”哪一层最明显

## 3. 分级建议

### P1：立即处理
满足任一项即可先按高优先级推进：

- [ ] 已明确影响生产业务
- [ ] control-plane / apiserver / 核心监控链路异常
- [ ] 多节点、多组件持续异常重启
- [ ] 告警指向容量打满、服务不可用、关键证书/认证异常

### P2：尽快跟进

- [ ] 巡检报告指出高风险，但当前业务尚未受损
- [ ] 某关键组件反复异常，但具备冗余
- [ ] 告警内容真实，但影响范围有限
- [ ] 指标异常持续存在，需要进一步核实根因

### P3：观察与优化

- [ ] 周期性波动，暂未见业务影响
- [ ] 单个组件偶发重启，且无法稳定复现
- [ ] 巡检项更像优化建议，不是立即故障
- [ ] 客户关注的是风险说明或后续治理建议

### P4：展示/模板/规则类

- [ ] 告警有触发，但 summary/通知内容不理想
- [ ] 日志后端正常，只有 UI 展示异常
- [ ] 指标、日志、告警本体正常，但页面体验有偏差
- [ ] 巡检/告警规则阈值需要调优

## 4. 现场最小证据清单

### 如果是巡检 / 告警
- [ ] 告警名称 / 巡检项名称
- [ ] 首次发生时间、最近发生时间
- [ ] 影响对象（节点 / Pod / 集群 / 组件）
- [ ] 当前值、阈值、持续时长
- [ ] 是否已有业务侧反馈

### 如果是组件异常重启
- [ ] 重启对象名称
- [ ] 重启次数与时间窗口
- [ ] 是否集中在某节点或某命名空间
- [ ] 同时段是否有资源、事件、变更

### 如果是日志 / 展示异常
- [ ] `kubectl logs` 或底层查询是否正常
- [ ] 问题是否只在页面出现
- [ ] 是否与超长日志、筛选条件、时间窗有关

## 5. 标准处置路径

### A. 更像真实故障
- [ ] 回到对应 runbook 按现象排障
- [ ] 先保留现场证据，再决定是否重启/变更
- [ ] 明确影响范围与优先级

### B. 更像规则/模板问题
- [ ] 核对规则阈值
- [ ] 核对通知模板字段
- [ ] 判断是否要优化告警噪声

### C. 更像展示/UI 问题
- [ ] 先验证后端采集与查询正常
- [ ] 保留截图、时间点、筛选条件
- [ ] 与产品/UI 问题分流处理

## 6. 支持现场一句话口径

- 如果是高优先级：
  - 当前信号已具备真实影响或高风险特征，建议按故障先处理，再回头做规则优化。
- 如果是中优先级：
  - 当前更像持续异常或潜在风险，建议尽快收集证据并安排进一步排查。
- 如果是低优先级：
  - 当前暂未看到直接业务影响，更适合持续观察并结合后续变化判断是否升级处理。
- 如果是展示/模板类：
  - 当前更像告警内容、规则阈值或展示链路问题，本体能力未必异常，可按配置优化方向推进。


## 6.1 告警邮件 / SMTP 投递专项检查

当问题是“告警邮件是否已发送到 SMTP 服务器”时，先不要只看 Alertmanager 或只问 SMTP 对端，按下面四层拆：

- [ ] **规则层**：告警是否处于 firing，时间窗口与用户反馈是否一致。
- [ ] **策略层**：通知策略是否匹配该告警，联系人/接收人是否包含邮件渠道。
- [ ] **投递层**：查看 courier-api 日志，确认是否有对应时间窗口的 SMTP 发送记录或错误。
- [ ] **对端层**：SMTP 服务器日志是否收到连接，是否因认证、TLS、发件人策略、垃圾/隔离策略拒收。

判断口径：

- courier-api 无投递记录：优先回查通知策略匹配与联系人配置。
- courier-api 有失败记录：按 SMTP 地址、端口、TLS、认证、网络连通性分层排查。
- courier-api 显示投递成功但对端未收：对齐 SMTP 服务器日志和邮件策略，不要继续重启平台组件。

参考案例：

- [TICKET-1351508304](../../ticket-documents/cases/TICKET-1351508304%20环境内配置了邮件服务器，需要确认下是哪个组件将告警邮件发送到邮件服务器.md)：courier-api 是平台侧确认 SMTP 投递的最小观测点。
