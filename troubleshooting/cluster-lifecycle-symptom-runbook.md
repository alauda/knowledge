# ACP 集群与节点生命周期现象排障手册

> 这份手册按支持现场最常见的**现象**组织，不按平台功能模块组织。
>
> 目标：先判断问题更像卡在 **主机 / 节点组件 / 集群接入 / 平台展示** 哪一层，再做最小验证。

## 1. 首页分诊

先把问题分成四层：

1. **主机层**
   - 机器是否存活
   - IP 是否冲突
   - 时间是否同步
   - 磁盘是否满
   - 网络是否异常
2. **节点组件层**
   - kubelet
   - container runtime
   - 证书
   - systemd 服务
3. **集群接入层**
   - 加节点
   - 纳管
   - 替换
   - 初始化脚本
   - 二进制/镜像获取与解压
4. **平台展示层**
   - 页面状态
   - 告警
   - 调度结果
   - 节点角色展示

### 最短判断顺序

1. 节点机器本身是否正常
2. 节点 IP / DNS / 时间 / 磁盘 是否异常
3. kubelet / runtime 是否正常
4. 节点是否真的成功加入或重新加入集群
5. 平台侧显示是否只是滞后或表象

---

## 2. 现象：添加节点失败 / `EnsureContainerRuntime` 报错

### 更像卡在哪一层
优先怀疑：**集群接入层**，其次是主机层与 runtime 层。

### 最小验证动作

1. 确认节点架构、OS 与目标集群要求一致
2. 确认下载的二进制包是否完整
3. 检查节点到制品源/控制面的网络质量
4. 检查 `/tmp`、`/var/lib` 等路径的空间与权限
5. 在目标节点手动验证解压/读取是否正常
6. 再回头看 runtime 版本与安装状态

### 常见根因

- 带宽或网络质量差，导致包下载不完整
- 压缩包内容异常，解压失败
- 节点目录权限、空间、清理策略有问题
- 节点与集群架构不一致
- runtime 前置条件不满足

### 代表案例

- [TICKET-1295043794](../../ticket-documents/cases/TICKET-1295043794%20业务集群添加节点EnsureContainerRuntime报错.md)：最终指向包解压/传输异常
- [TICKET-1336266904](../../ticket-documents/cases/TICKET-1336266904%20集群中添加节点报错，根据错误提示无法定位出原因.md)：添加节点报错但错误提示不足

## 3. 现象：节点 `NotReady` / 节点异常 / 节点不可用

### 更像卡在哪一层
优先怀疑：**主机层** 与 **节点组件层**。

### 最小验证动作

1. 确认节点机器是否真的重启/宕机
2. 确认节点 IP 是否被复用或冲突
3. 检查 kubelet / runtime 状态
4. 查看节点磁盘、`/var/lib`、镜像目录是否异常
5. 检查证书、时间同步、网络连通
6. 必要时清理旧节点信息，再观察重新加入结果

### 常见根因

- IP 与旧主机冲突
- kubelet 假性重启或状态异常
- runtime 或底层容器目录异常
- 节点网络切换后 Pod 间网络不通
- 节点磁盘压力过高

### 代表案例

- [TICKET-1321672154](../../ticket-documents/cases/TICKET-1321672154%20客户方两个集群共计四个节点出现不可用问题.md)：旧机器占用原 IP，清理后恢复
- [TICKET-1294192034](../../ticket-documents/cases/TICKET-1294192034%20集群节点notready.md)：节点 `NotReady`
- [TICKET-1345186394](../../ticket-documents/cases/TICKET-1345186394%2010多个节点挂掉，导致不可用.md)：多个节点掉线导致不可用
- [TICKET-1351047364](../../ticket-documents/cases/TICKET-1351047364%20节点状态异常.md)：节点异常且 SSH 已不可达时，先抬到宿主机 / IaaS 层，主机重启后恢复只是闭环动作，不等同于 kubelet 根因。

## 4. 现象：节点替换 / 节点重新加入后行为异常

### 更像卡在哪一层
这是**集群接入层**和**主机层**交叉问题。

### 最小验证动作

1. 确认是“原节点恢复”还是“新节点替换旧节点”
2. 确认旧节点是否已彻底下线
3. 确认节点名、IP、角色、标签、污点是否延续正确
4. 检查平台或集群中是否残留旧节点身份信息
5. 检查新节点网络、证书、runtime、挂载能力

### 常见根因

- 旧节点未彻底退出，身份冲突
- 换机后 IP/主机名/角色与原状态不一致
- 新节点基础能力不满足旧节点工作负载要求

### 代表案例

- [TICKET-1347683424](../../ticket-documents/cases/TICKET-1347683424%20集群节点替换.md)：集群节点替换
- [TICKET-1283364834](../../ticket-documents/cases/TICKET-1283364834%20global集群有3个节点，现在需要关闭一台节点增加磁盘，会有哪些影响？重启后是否会自动加入集群？.md)：节点下线扩盘与重入影响评估

## 5. 现象：客户问“这个节点 / 架构 / OS 能不能纳管”

### 更像卡在哪一层
优先是**兼容性与前置依赖确认**问题，不应直接按故障单处理。

### 最小验证动作

1. 确认 CPU 架构
2. 确认 OS 发行版与版本
3. 确认是否依赖特定内核模块
4. 确认 GPU、特殊网卡、特殊 runtime 的额外要求
5. 再判断是否可纳管或是否需要特殊方案

### 常见根因

- 现场把“理论兼容”理解成“现网可直接接入”
- 忽略内核模块、驱动、runtime 依赖
- 异构节点接入缺少额外验证

### 代表案例

- [TICKET-1263386154](../../ticket-documents/cases/TICKET-1263386154%20在添加redhat8节点时的init脚本是否和redhat7.9的通用.md)
- [TICKET-1264991094](../../ticket-documents/cases/TICKET-1264991094%20平台或集群组件是否有依赖nf_tables模块.md)
- [TICKET-1298138764](../../ticket-documents/cases/TICKET-1298138764%20ACP是否支持纳管海光C86架构+欧拉系统节点.md)
- [TICKET-1292623834](../../ticket-documents/cases/TICKET-1292623834%20业务K8S集群未升到3.18版本，是否还能新添加GPU节点服务器.md)

## 6. 首页速查：支持现场优先问这 6 个问题

1. 这是普通 worker 还是关键节点？
2. 是节点恢复，还是节点替换？
3. 节点 IP / 主机名 / 角色 有没有变化？
4. kubelet / runtime 是否正常？
5. 节点本机磁盘、时间、网络是否正常？
6. 是平台页面异常，还是节点实际真的不可用？
7. 升级 / 加节点场景下是否先满足时间同步与 CA 记录一致性前置？

### 升级前置补充

- boot / upgrade Pod 报错时，先核对所有控制面和关键节点时间同步；参考 [TICKET-1351129734](../../ticket-documents/cases/TICKET-1351129734%20升级的boot所对应的Pod报错.md)。
- 升级报证书问题时，不要只判断“证书过期”，要比对 `clusters.clusterregistry.k8s.io` 中记录的 CA 与节点实际 CA 链；参考 [TICKET-1350997244](../../ticket-documents/cases/TICKET-1350997244%20集群升级报错证书问题.md)。

## 7. 常见误判速查

- **页面异常 ≠ 节点一定异常**
- **加节点失败 ≠ 平台逻辑一定有 bug**
- **节点不可用 ≠ 机器一定重启过**
- **节点替换 ≠ 普通重启**
- **纳管支持咨询 ≠ 可以跳过前置依赖检查**

## 运行时仓库配置与 11443 镜像拉取（2026-05-31）

来自 [[TICKET-1309777894 bug，偶发kubelet无法拉取11443镜像仓库]] 的补充：

- kubelet 偶发拉取内置 `11443` 仓库失败时，不要先默认是平台 registry 或网络抖动。
- 先核对 containerd 的 CRI registry 配置是否显式存在，尤其是 `/etc/containerd/config.toml` 中 `plugins."io.containerd.grpc.v1.cri".registry` 相关段。
- 若配置缺失或未被托管链重载，应先修运行时仓库配置，再继续排查证书、仓库服务和网络连通性。

## 9. 现象：global /cpaas 挂载异常导致平台组件大量异常（2026-06-03）

来自 [[TICKET-1264184764 平台组件无法访问]] 的补充：

- 平台组件大量异常或无法访问时，除了查 Pod/控制面服务，还要先确认 global 节点 `/cpaas` 是否正常挂载。
- 若单节点文件系统数据异常导致 `/cpaas` 挂载失败，先保护磁盘与挂载证据；`xfs_repair -L` 属有风险修复动作，需确认设备、备份和维护窗口后执行。
- 修复挂载后继续检查其他节点 `/cpaas` 与根目录水位；本案还叠加了 2 节点 `/cpaas` 满、1 节点根目录约 91%，清理空间后集群恢复。
- 不要把“组件大量异常”直接写成 apollo/API/前端故障；控制面磁盘、挂载和文件系统异常可能是上游根因。

## 10. 现象：告警风暴 / etcd 频繁切主 / ARS 卡住的聚合判断（2026-06-06）

来自 [[TICKET-1300123204 很多集群的告警]]、[[TICKET-1320970014 mazk集群频繁etcd切主，部分节点监控显示异常]]、[[TICKET-1270569724 生产集群升级 ars 卡主]] 的补充：

- 多告警同时出现时，先按节点、资源类型、时间窗和恢复动作聚合；不要逐条拆成多个组件故障。若增加节点内存后整体恢复，更稳落点是节点内存前提不足。
- etcd 频繁切主叠加部分节点监控异常时，先看 etcd slow request / leader change 与磁盘低延迟随机写能力；机械盘顺序吞吐不能替代 fsync/4K 随机写证据。
- 升级 ARS 卡住且 sentry 报 registry timeout 时，必须用节点侧 `helm login` / `helm pull` 做反证；若节点侧正常而 sentry 内部超时，优先处理 sentry 运行态并保留重启前日志。

## 11. 现象：节点 IP / OS / Docker / 端口前置与集群重建咨询（2026-06-19）

来自 medium + merged-existing/no-case 工单的稳定补充：这类问题多是安装/升级/下线前置边界，不要直接当故障 RCA。

### 最小判断链

1. **节点 IP 调整**：业务未投产或可接受数据丢失时，优先建议删除后重建集群；逐个替换 IP 需要完整控制面、证书、节点身份和网络验证，不应作为默认低风险路径。
2. **OS / 内核兼容**：同一发行版内核大版本一致时，一般不需要为小版本反复适配；但 Rocky Linux 10.1 等未适配 OS 不能承诺加节点可用。
3. **预装 Docker**：已有 Docker 的节点用于 ACP 安装时，应先卸载并清理干净，再由平台安装/托管 runtime，避免版本、配置与服务管理冲突。
4. **节点网络管理服务**：产品没有强制推荐 NetworkManager 或 systemd-networkd；但功能测试基线偏 NetworkManager，现场优先按已验证基线评估。
5. **业务集群端口**：ACP 4.x 不再使用 11781，开放 6443 / 11780 等当前版本所需端口即可进入创建验证。
6. **全量关停 / 下线 / 重建**：关机前先备份 etcd；下线前先确认业务迁移、Ceph/TopoLVM/ALB/插件资源清理与 cleanup 卡点；重建前确认监控、日志、计费等页面配置是否需要重新配置。
7. **升级 checklist / rpch**：先区分误报、可忽略项和升级后需补配置项；不要把所有 checklist error 都升级成产品故障。

- TICKET-1349278474：升级到 4.3 前按 apps.yaml / extension packages 完整性确认可升级物料。
- TICKET-1349537474：生产集群重建时，监控/日志/计费页面配置是否保留需提前确认，已允许数据丢失时仍要提示配置重建成本。
- TICKET-1349575154：3.x 升 4.x 后“集群-插件”页面下线，入口迁移到“应用商店管理-集群插件”。
- TICKET-1350185574：4.2.4 后中文界面需部署中文包插件。
- TICKET-1350215634：整套 K8s 集群关机前先做 etcd 备份，再按窗口统一关机/开机验证。

## 11.1 升级路径、OS 适配、公网 kubeconfig 与语言包短 FAQ（2026-06-20）

### 适用场景

- 客户询问 4.2.4 是否可直接升级到 4.3.1；
- 客户询问 Ubuntu 24.04、TencentOS v3、EulerOS/KY 等 OS 是否在当前版本支持矩阵内；
- 需要导出公网访问 kubeconfig；
- 升级到 4.2.4 后页面无法切换中文。

### 排查路径

1. 升级路径先查目标版本 release/upgrade 口径；当前样本中 4.2.4 可直接升级到 4.3.1，但生产答复仍需按目标版本发布说明确认。
2. OS 支持只按已认证矩阵答复：Ubuntu 24.04、TencentOS v3 等未适配系统不能承诺可直接部署/纳管，需要项目或产品认证评估。
3. 公网 kubeconfig 生成时，先从业务集群页面导出 kubeconfig，再把内部 token 替换为实际用户个人 token，并将 server 地址改为公网入口；随后验证 RBAC、证书和网络可达性。
4. 升级后中文界面不可切换时，先确认中文包插件是否已部署；不要直接判断为前端国际化故障。

### 处理建议

- 对升级路径咨询，给“当前口径 + 仍需按版本文档/升级工具前置检查”的边界。
- 对 OS 适配咨询，不用“理论兼容”替代“产品认证”；未认证 OS 走 PM/产品认证或实验验证。
- 公网 kubeconfig 属访问入口与身份替换操作，生成后必须做最小 kubectl 验证，并确认 token 权限范围。
- 语言包问题优先检查插件安装状态和版本匹配。

### 风险边界

- 直升路径会随目标补丁版本、扩展包和 apps.yaml 状态变化，不应脱离现场前置检查。
- 未认证 OS 可能在内核、runtime、网络插件、驱动上失败。
- 公网 kubeconfig 暴露面更大，token 应使用个人身份且按最小权限管理。

### 关联工单

- TICKET-1351688214：4.2.4 到 4.3.1 升级路径咨询，样本答复为可直接升级。
- TICKET-1348661524：3.18.2/4.2.x 对 Ubuntu 24.04 未适配，需单独适配认证。
- TICKET-1349019614：操作系统兼容咨询，TencentOS v3 等不在产品基线支持范围时需走认证评估。
- TICKET-1349821804：公网 kubeconfig 通过导出后替换个人 token 与公网 server 地址验证。
- TICKET-1350185574：4.2.4 后无法切换中文，通过部署中文包插件处理。

### Deep-case 信号

- 当前判断：不需要。
- 原因：本组属于升级/OS/访问配置/插件前置 FAQ，证据链短且口径稳定。
- 还缺什么证据：若要 deep-case，需要升级前置检查结果、目标版本 release note、OS 认证矩阵、kubeconfig 访问失败报错、语言包插件状态与前端请求日志。

## 12. 现象：部署节点权限、扩容后资源未刷新、组件临时存储限制（2026-06-19）

来自 medium + merged-existing/no-case 工单的稳定补充：这些问题常落在**主机前置权限、kubelet 节点状态刷新、已知资源阈值缺陷**三类，不要直接扩写成完整故障 RCA。

### 最小判断链

1. **部署节点权限**：安装/纳管节点需要能创建 `/var/lib/containerd/`、`/var/lib/kubelet/`、`/var/lib/etcd/`、`/cpaas/` 等目录，并执行分区、挂载、systemd、runtime 安装/重启等操作；当前应明确要求 root 或具备等价 sudo 权限的用户。
2. **宿主机 CPU 扩容后集群未感知**：先确认主机侧 CPU 已扩容并被 OS 识别，再通过 `systemctl daemon-reload` 和维护窗口内重启 `kubelet` 触发节点 allocatable 刷新；不要只刷新页面或等待平台自动发现。
3. **平台组件历史异常 Pod / ephemeral-storage 100Mi**：若 `metis` 等组件出现历史 `ContainerStatusUnknown`，事件指向 ephemeral storage 超 100Mi，优先按版本已知缺陷/资源限制不合理归类，并收集事件与版本信息；不要误判为业务存储或节点磁盘已满。
4. **DR / Web CLI 令牌失效类问题**：Web CLI 间歇失败且日志出现 `invalid access token: authentication head is invalid` 时，优先按登录态/token 失效边界处理；复现时再补 web-terminal、认证服务与浏览器会话时间线。

### 关联工单

- TICKET-1353009894：部署集群过程中客户只能提供非 root 用户，需说明目录、分区、runtime、systemd 等必须具备 root/sudo 权限。
- TICKET-1353356474：节点 CPU 扩容后集群侧资源未变化，执行 `daemon-reload` 并重启 kubelet 后恢复可见。
- TICKET-1353316694：业务集群 `metis` 历史异常 Pod 事件显示 ephemeral-storage 超 100Mi，归入组件临时存储限制已知缺陷/优化。
- TICKET-1353352114：DR 环境 Web CLI 间歇不可用，日志显示 access token 认证头异常，恢复后需等待复现补证。

### Deep-case 信号

- 当前判断：多数不需要。
- 原因：节点权限、kubelet 刷新、已知 limit 缺陷和登录态失效都属于短链路 FAQ/边界说明。
- 还缺什么证据：若要 deep-case，需要安装/扩容/访问的完整时间线、节点 OS 与 kubelet 日志、Pod event 与组件版本、web-terminal/认证服务日志、以及恢复前后 diff。


## 12.1 global worker 迁移、节点删除中断与 Calico 网卡探测边界（2026-06-22）

### 适用场景

- global worker 节点从 VMware 迁移到其他虚拟化平台；
- 节点送修/关机期间曾在平台执行删除，恢复后又自动回到集群；
- 虚拟机迁物理机或迁移后网卡名变化，例如 `ens192` 变为 `ens3` / `bond0`；
- 软负载 `haproxy + keepalived` 切换到硬负载并保持 VIP 不变。

### 排查路径

1. 节点迁移先确认 IP、主机名、磁盘路径、数据目录和虚拟化资源是否保持不变；只迁底层虚机且 IP 不变时，主线是节点停止调度、驱逐、关机迁移和恢复验证。
2. global worker 迁移若涉及日志/ES/nevermore 等固定节点组件，应先新上架 worker，再停止调度旧 worker，卸载/迁移日志采集与日志存储插件，驱逐 Pod 后再重装到新节点。
3. 网卡名变化时，Calico 场景需核对 `IP_AUTODETECTION_METHOD`；可用 `can-reach` 或 interface 正则覆盖新旧网卡名，避免 Calico 选错地址。
4. 删除节点过程中服务器关机或删除中断，修复后节点自动回来不代表状态完全干净；建议重新执行删除、跑清理脚本和初始化脚本后再按流程加回。
5. 软负载切硬负载时，除停 keepalived/切 VIP 外，还需评估并重启所有节点 kubelet/kube-proxy，使控制面访问路径和 service/proxy 状态收敛。

### 风险边界

- global worker 上的日志存储、采集组件和本地盘可能有节点绑定，不是简单关机迁移。
- 修改 Calico 网卡探测会影响 Pod 网络地址选择，需在业务集群逐一确认 DaemonSet 与节点地址。
- 节点删除/清理属于变更动作，直接让返修节点继续运行可能遗留证书、runtime、Pod 或平台状态问题。
- VIP/负载切换需有测试环境演练、回滚计划和控制面连通性验证。

### 关联工单

- TICKET-1353964004：global worker 从 VMware 迁移到其他虚拟化平台，需新上架 worker、迁移日志/ES/nevermore 组件并关注 Calico 网卡名变化。
- TICKET-1353969524：虚拟机 master 替换物理机时，历史 rpch 使用 `IP_AUTODETECTION_METHOD=can-reach`，网卡名不一致可通过 Calico 自动探测适配。
- TICKET-1353969174：节点故障删除过程中服务器关机，修好后自动加回；建议重新删除、清理并初始化后再加回。
- TICKET-1353957834：软负载切硬负载需停 keepalived、切 VIP，并重启所有节点 kubelet/kube-proxy。

### Deep-case 信号

- 当前判断：可选。
- 原因：这是节点迁移/删除/负载切换操作边界，通常可按变更 runbook 处理。
- 还缺什么证据：若要 deep-case，需要迁移拓扑、组件清单、Calico DS diff、节点删除任务状态、清理脚本输出、VIP 切换窗口、kubelet/kube-proxy 日志与业务影响时间线。

## 12.2 内部网络故障、外部 VIP 与节点删除阻断短 FAQ（2026-06-21）

### 适用场景

- 客户内部网络故障预计持续较久，咨询 ACP/global/workload/DCS 等组件是否需要关机以避免时间偏移或状态异常；
- 海外/新 global 部署卡住，现场使用自建 VIP，但部署要求外部 VIP；
- 页面无法删除节点，节点上存在异常 Pod 或阻断器，需要判断是否可强删。

### 排查路径

1. 网络故障咨询先区分控制面/业务面/时间同步源：NTP、HLB、DCS、global 与 workload 是否会同时失联。
2. 长时间网络不可用且无法保障时间同步时，按现场确认的停机顺序处理；样本中更稳妥口径为先关虚拟机，再关物理机。
3. 新 global 部署卡住先核部署拓扑与 VIP 类型，海外环境或特定方案要求外部 VIP 时，不要用自建 VIP 继续反复重试。
4. 节点删除失败先查看阻断器、Pod 驱逐、finalizer/保护对象和节点角色；删除阻断器属于受控恢复动作，应保留对象和原因。

### 风险边界

- 关机/停机是变更动作，需业务窗口、数据一致性和恢复顺序确认。
- 自建 VIP 与外部 VIP 不是简单字段替换，可能影响入口可达性、证书、HA 与部署前置检查。
- 强制删除阻断器或节点保护对象可能导致工作负载中断或状态残留。

### 关联工单

- TICKET-1355335594：客户内部网络故障持续，现场确认先关虚拟机再关物理机更稳妥。
- TICKET-1355415704：新 global 环境部署卡住，使用自建 VIP 与海外环境外部 VIP 要求不符，后续改用外部 VIP。
- TICKET-1354426384：页面无法删除节点，节点存在阻断器，需等待进一步分析或受控删除阻断器。

### Deep-case 信号

- 原因：本组是变更/部署前置和节点删除保护边界，适合短 FAQ。
- 还缺什么证据：若要 deep-case，需要部署拓扑、VIP 规划、安装日志、节点对象/阻断器 YAML、Pod 驱逐事件、停机/恢复时间线和业务影响记录。
