# 自定义角色用户切换到新建集群后查看资源无权限

## 现象
- 绑定自定义角色的用户切换到新建集群后，查看 `PV` 等资源时提示无权限。
- 现场排查表明，问题并非用户未绑定角色，而是角色绑定后的权限未正确覆盖升级后新建的集群范围。
- 问题发生背景与 `v4.2` 升级后自定义角色规则渲染异常有关，后续在 `v4.3` 场景中也可能出现同类现象。

## 排查过程与命令
- 首先检查用户角色绑定情况，确认用户已绑定自定义角色，但对应角色未加载升级后新建集群名称，因此登录用户实际未获得该集群上的资源访问权限。
- 为避免直接修改造成配置不可回滚，先对客户环境中的自定义角色及其绑定关系做备份，备份路径为 `global-master1:/root/roletemplate-update-fix-4.2.4`。
- 按实际角色名称设置变量，并生成统一时间戳，便于后续批量修复与审计追踪：
  ```bash
  TS=$(date -u +%FT%TZ)
  echo "$RT========$TS"
  ```
- 备份与该自定义角色相关的 `userbinding`：
  ```bash
  kubectl get userbinding -l auth.cpaas.io/role.name=$RT -oyaml > /root/roletemplate-update-fix-4.2.4/$RT-userbinding.yaml
  ```
- 备份对应的 `roletemplate`：
  ```bash
  kubectl get roletemplate $RT -oyaml > /root/roletemplate-update-fix-4.2.4/roletemplate-$RT.yaml
  ```
- 随后对 `roletemplate` 补充必要标记，并通过注解触发规则重新渲染：
  ```bash
  kubectl label roletemplate $RT auth.cpaas.io/roletemplate.frozen=false --overwrite
  kubectl annotate roletemplate $RT auth.cpaas.io/bootstrap-fix="$TS" --overwrite
  ```
- 在 `roletemplate` 重新渲染后，继续对绑定该角色的 `userbinding` 补充注解，触发用户绑定关系重新渲染：
  ```bash
  kubectl get userbinding -l auth.cpaas.io/role.name=$RT -o name | xargs -I {} kubectl annotate {} auth.cpaas.io/bootstrap-fix="$TS" --overwrite
  ```
- 经过上述处理后，用户重新获得新建集群对应权限，查看 `PV` 等资源恢复正常。由此可以确认，问题核心在于升级后自定义角色规则未自动覆盖新建集群，而不是用户绑定关系本身缺失。

## 根因与修复方案
- **根因**
  - 升级到 `v4.2` 后，部分自定义角色缺少必要 label，导致 `roletemplate` 与 `userbinding` 未正确完成新建集群相关权限规则渲染。
  - 用户虽然绑定了自定义角色，但切换到新建集群后无法获得对应资源权限，最终表现为查看 `PV` 等资源时报无权限。
- **临时缓解方案**
  - 先备份受影响的 `roletemplate` 与 `userbinding`。
  - 为目标 `roletemplate` 补充 `auth.cpaas.io/roletemplate.frozen=false` label，并添加 `auth.cpaas.io/bootstrap-fix` 注解触发重新渲染。
  - 为相关 `userbinding` 添加 `auth.cpaas.io/bootstrap-fix` 注解，触发绑定关系重新渲染。
- **根本解决方案**
  - 修复升级后自定义角色缺少必要 label 的生成或迁移逻辑，确保新建集群能够自动纳入权限规则渲染范围。
  - 将 `roletemplate` / `userbinding` 的升级后自检与补偿机制纳入产品默认流程，避免后续版本再次出现权限未生效问题。
