# ETCD 数据库使用量超过预警阈值

## 现象
- 巡检触发 `cluster.kube.etcd.db.size.mb.per.instance` 指标告警，表现为 ETCD 数据库使用量超过预警阈值。
- 现场确认该指标对应的查询语句为：
  ```promql
  etcdmvccdbtotalsizeinbytes{namespace="kube-system"} / 1024 / 1024
  ```
  单位为 MB。
- 告警期间 etcd 服务状态正常，未发生切主，但数据库空间占用持续偏高。

## 排查过程与命令
- 首先通过 `etcdctl endpoint status` 确认 etcd 集群状态正常，判断本次问题不属于成员异常或主从切换导致的可用性问题。
- 随后确认 etcd 数据库大小，并继续判断空间增长是由热点 key 持续刷写，还是由历史数据累积导致。
- 为了统计实际 key 分布，执行以下命令：
  ```bash
  etcdctl --cacert /etc/kubernetes/pki/etcd/ca.crt --cert /etc/kubernetes/pki/etcd/server.crt --key /etc/kubernetes/pki/etcd/server.key --endpoints https://localhost:2379 get / --prefix --keys-only | grep -v ^$ | awk -F '/' '{ h[$3]++ } END {for (k in h) print h[k], k}' | sort -nr
  ```
- 统计结果显示，频繁变动的 key 没有特别大的值，etcd 按 key 分析得到的实际占用并不高，但节点侧观察到的磁盘占用明显偏大。基于这一现象，可以初步排除“单个或少量热点 key 异常膨胀”这一方向。
- 继续检查平台业务集群事件，发现近 7 天内累计存在数万条事件记录。结合前述现象，判断问题更接近于历史 revision 或事件类数据累积带来的空间占用增长。
- 随后执行 etcd 压缩操作，压缩完成后 etcd 空间占用下降至约 `50MB`，说明历史 revision 回收后空间占用得到明显释放，进一步支持“历史数据未及时回收”这一判断。
- 为了确认告警口径及后续巡检项定位方式，现场进一步核对了巡检指标来源，可通过以下命令查找对应 `indicator` 和 `query`：
  ```bash
  kubectl get inspectionitems.ait.alauda.io -A | grep <keyword>
  kubectl get inspectionitems.ait.alauda.io cluster-etcd-database-usage -oyaml | grep indicator
  for i in $(kubectl get cm -n cpaas-system | awk '{print $1}'); do result=$(kubectl get cm -n cpaas-system $i -oyaml | grep <indicator_name>); if [ -n "$result" ]; then echo $i; fi; done 2>/dev/null
  kubectl edit cm -n cpaas-system indicators-standard-cluster
  ```
- 综合以上排查结果，可以确认：本次告警与 etcd 服务异常无关，核心问题在于数据库历史数据未及时压缩回收，导致空间指标持续升高。

## 根因与修复方案
- **根因**
  - 历史 revision 未及时回收，导致 etcd MVCC 数据持续累积，最终表现为数据库空间占用升高并触发 `cluster.kube.etcd.db.size.mb.per.instance` 告警。
- **临时缓解方案**
  - 执行 etcd 压缩操作，回收历史 revision 占用空间。
- **根本解决方案**
  - 建立针对 etcd 历史 revision 与事件数据增长的持续治理机制，避免长期累积再次推高数据库空间占用。
  - 在巡检侧保留并定期核对 `cluster.kube.etcd.db.size.mb.per.instance` 对应指标与查询配置，确保告警口径清晰可追溯。
