# FAQ / Troubleshooting 文档说明

这批文档对应的是售后知识库里的两层内容：

- `faq/` 更偏向 **产品知识层**，回答“是什么、支不支持、边界在哪、为什么会这样”。
- `troubleshooting/` 更偏向 **经验知识层**，回答“遇到问题先怎么看、怎么分诊、怎么排、怎么止血”。

这和项目总 README 里的思路一致：先保留目录职责，再把高价值知识沉淀成可检索、可复用、可回链的 Markdown 资产。

## `faq/` 目录的价值

`faq/` 里的文档主要承担“产品知识本体”的作用，适合放这些内容：

- 常见问题解释
- 产品能力边界
- 版本差异和支持范围
- 用户最容易误解的概念说明
- 从工单中抽出来的高频问答

这类文档的特点是：

- 更适合给一线支持快速答复客户
- 更适合做机器人召回的标准问答入口
- 更适合把“能不能做、为什么不行、边界是什么”说清楚
- 不强调完整工单过程，而强调结论和口径统一

对后续知识库建设的价值在于：

- 可以作为产品知识的稳定入口
- 能减少重复解释和口径不一致
- 方便和 `kb/` 里的排障文档互相跳转
- 适合后续逐步统一成公司标准 Troubleshooting / FAQ 模板

## `troubleshooting/` 目录的价值

`troubleshooting/` 里的文档主要承担“经验知识层”的作用，适合放这些内容：

- 排障手册
- checklist / quick card
- 症状型 runbook
- 现场分诊步骤
- 复盘后的判断路径

这类文档的特点是：

- 更强调现场可执行
- 更强调按症状分流，而不是按组件堆说明
- 更适合支持同事第一时间定位问题层级
- 更适合沉淀“先看什么、再看什么、不要先误判什么”

对后续知识库建设的价值在于：

- 可以把零散工单经验收敛成标准排障路径
- 可以作为机器人辅助诊断的知识底座
- 可以持续沉淀通用判断路径，减少重复踩坑
- 可以逐步从工单材料过渡到标准 Troubleshooting 文档

## 当前阶段说明

现阶段这批文档仍处在“先沉淀、再规范”的过渡阶段，所以正文里还会保留少量工单信息、案例线索和排查过程。

后续我们会尽量向公司标准格式收敛，逐步做到：

- 统一英文标题和章节结构
- 补齐标准 frontmatter，明确 ACP 版本和业务线分类
- 将工单过程信息逐步收口到案例层或关联信息中
- 让 `faq/` 更偏产品解释，让 `troubleshooting/` 更偏排障执行

当前版本的核心价值是：先把售后部门真正有用的经验沉淀下来，再逐步整理成更标准、更适合平台化复用的知识资产。
